## 一、E2B 是什么？

E2B（e2b.dev）是专为 AI Agent 设计的**云端代码执行沙箱**服务，基于 Firecracker microVM（AWS Lambda 也使用同款）。

```
┌──────────────────────────────────────────────────────┐
│                  E2B 架构                             │
│                                                      │
│  你的 LangChain Agent                                │
│          │  API 调用（HTTPS）                         │
│          ▼                                           │
│    E2B API 服务                                      │
│          │                                           │
│          ▼                                           │
│  ┌───────────────────┐                              │
│  │   Firecracker VM  │  ← 每次调用独立 microVM      │
│  │  （100ms 启动）   │                              │
│  │   完整 Linux 内核  │                              │
│  │   你的代码在这里   │                              │
│  └───────────────────┘                              │
└──────────────────────────────────────────────────────┘
```

**Firecracker vs Docker 的关键区别：**

```
Docker 容器：
  - 共享宿主机内核（Namespace 隔离）
  - 内核漏洞可能导致容器逃逸
  - 启动：~500ms

Firecracker microVM：
  - 独立虚拟机，完整隔离的内核
  - 几乎无法逃逸
  - 启动：~100ms（专门优化过）
  - 内存开销：~5MB per VM（vs Docker ~100MB）
```

## 二、E2B 基础用法

```bash
pip install e2b-code-interpreter
```

### 2.1 简单执行

```python
from e2b_code_interpreter import Sandbox

# 创建沙箱（自动选择最近的数据中心）
with Sandbox(api_key="e2b_xxx") as sandbox:
    result = sandbox.run_code("print(1 + 1)")
    print(result.text)   # "2"
    print(result.error)  # None（无错误）
```

### 2.2 完整执行结果结构

```python
with Sandbox() as sandbox:
    result = sandbox.run_code("""
import pandas as pd
import numpy as np

df = pd.DataFrame({"x": range(10), "y": np.random.randn(10)})
print(df.describe())
print("shape:", df.shape)
""")
    
    # 文本输出
    print(result.text)          # "       x         y\ncount..."
    
    # 结构化输出
    for output in result.results:
        print(output.type)      # "text" / "image/png" / "error"
        if output.type == "image/png":
            print(output.data)  # base64 编码的图片
    
    # 错误信息
    if result.error:
        print(result.error.name)    # "NameError"
        print(result.error.value)   # "name 'x' is not defined"
        print(result.error.traceback)
```

### 2.3 Jupyter 内核模式（状态持久）

E2B 运行在 Jupyter 内核中，**变量在多次调用之间保持**。

```python
with Sandbox() as sandbox:
    # 第一次调用：定义变量
    sandbox.run_code("x = [1, 2, 3, 4, 5]")
    sandbox.run_code("import pandas as pd")
    
    # 第二次调用：变量仍然存在
    result = sandbox.run_code("print(sum(x))")
    print(result.text)  # "15"
    
    # 第三次调用：使用之前导入的库
    result = sandbox.run_code("pd.Series(x).describe()")
    print(result.text)
```

这对 Agent 的多步骤数据分析非常有价值——不需要每次重新加载数据。

## 三、流式输出

```python
with Sandbox() as sandbox:
    # 流式获取输出，适合长时间运行的代码

import time
for i in range(10):
    print(f"处理第 {i+1} 步...")

print("完成！")
```

## 四、文件系统操作

```python
with Sandbox() as sandbox:
    # 上传文件到沙箱
    with open("local_data.csv", "rb") as f:
        sandbox.files.write("/home/user/data.csv", f)
    
    # 在沙箱中处理文件
    result = sandbox.run_code("""
import pandas as pd
df = pd.read_csv('/home/user/data.csv')
print(df.head())
df_cleaned = df.dropna()
df_cleaned.to_csv('/home/user/output.csv', index=False)
""")
    
    # 从沙箱下载文件
    output_bytes = sandbox.files.read("/home/user/output.csv")
    with open("output_cleaned.csv", "wb") as f:
        f.write(output_bytes)
    
    print("文件已下载")
```

## 五、集成到 LangChain

### 5.1 基础 Tool 封装

```python
from e2b_code_interpreter import Sandbox
from langchain_core.tools import tool
import base64
import os

# 沙箱实例在 Agent 会话中复用（保持状态）
_sandbox: Optional[Sandbox] = None

def get_sandbox() -> Sandbox:
    global _sandbox
    if _sandbox is None or _sandbox.is_closed():
        _sandbox = Sandbox(
            api_key=os.environ["E2B_API_KEY"],
            timeout=300,      # 5分钟超时
        )
    return _sandbox


@tool
def e2b_run_code(code: str) -> str:
    """
    在云端隔离沙箱（E2B）中执行 Python 代码。
    
    特性：
    - 完全隔离的虚拟机环境
    - 状态持久：变量和导入在多次调用间保留
    - 可使用 matplotlib 生成图表（自动返回 base64 图片）
    - 预装库：numpy, pandas, matplotlib, scipy, sklearn, sympy
    
    Args:
        code: 要执行的 Python 代码
    """
    sandbox = get_sandbox()
    
    try:
        result = sandbox.run_code(code)
        
        if result.error:
            return f"❌ 错误 [{result.error.name}]：{result.error.value}\n{result.error.traceback[-500:]}"
        
        output_parts = []
        
        # 文本输出
        if result.text:
            output_parts.append(result.text)
        
        # 图片输出
        image_count = 0
        for item in result.results:
            if hasattr(item, "png") and item.png:
                image_count += 1
                # 保存到本地供后续查看
                img_path = f"/tmp/e2b_output_{image_count}.png"
                with open(img_path, "wb") as f:
                    f.write(base64.b64decode(item.png))
                output_parts.append(f"[图表已生成：{img_path}]")
        
        return "\n".join(output_parts) if output_parts else "代码执行完成（无输出）"
    
    except Exception as e:
        return f"❌ 沙箱执行异常：{str(e)}"


def cleanup_sandbox():
    """在 Agent 任务结束时关闭沙箱"""
    global _sandbox
    if _sandbox and not _sandbox.is_closed():
        _sandbox.close()
        _sandbox = None
```

### 5.2 带状态管理的 Agent

```python
from langchain_openai import ChatOpenAI
from langchain.agents import create_tool_calling_agent, AgentExecutor
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

llm = ChatOpenAI(model="gpt-4o", temperature=0)

SYSTEM = """你是一个数据分析专家 Agent，使用 E2B 云沙箱执行代码。

工作方式：
1. 沙箱状态在整个对话中保持，可以分步骤执行代码
2. 第一步加载数据，后续步骤直接使用已加载的变量
3. 生成图表时直接使用 matplotlib，图表会自动保存并返回路径

示例工作流：
- 步骤1：加载并探索数据
- 步骤2：清洗和预处理
- 步骤3：分析和可视化
- 步骤4：输出结论"""

prompt = ChatPromptTemplate.from_messages([
    ("system", SYSTEM),
    MessagesPlaceholder("agent_scratchpad"),
    ("human", "{input}"),
])

agent = create_tool_calling_agent(llm, [e2b_run_code], prompt)
executor = AgentExecutor(agent=agent, tools=[e2b_run_code], verbose=True)

try:
    result = executor.invoke({
        "input": "创建一个100行的随机销售数据，计算各月销售额，生成柱状图"
    })
    print(result["output"])
finally:
    cleanup_sandbox()
```

## 六、五种沙箱方案对比与选型

```
┌──────────────┬────────────┬────────────┬────────────┬──────────────┬──────────────┐
│    方案       │  隔离级别   │  启动延迟   │  维护成本  │    成本       │   适用场景    │
├──────────────┼────────────┼────────────┼────────────┼──────────────┼──────────────┤
│ subprocess   │ 低（进程）  │ <100ms     │ 低         │ 免费          │ 低风险代码   │
│ +resource    │            │            │            │              │ 内部工具      │
├──────────────┼────────────┼────────────┼────────────┼──────────────┼──────────────┤
│ subprocess   │ 中（进程+  │ <200ms     │ 中         │ 免费          │ 受信任用户   │
│ +seccomp     │ syscall）  │            │            │ （Linux 限定）│ 内部平台      │
├──────────────┼────────────┼────────────┼────────────┼──────────────┼──────────────┤

│              │            │            │            │              │ B 端产品      │
├──────────────┼────────────┼────────────┼────────────┼──────────────┼──────────────┤
│ E2B          │ 极高（VM）  │ ~100ms     │ 极低       │ 按量计费      │ C 端产品      │

├──────────────┼────────────┼────────────┼────────────┼──────────────┼──────────────┤
│ Firecracker  │ 极高（VM）  │ ~125ms     │ 高         │ 自建服务器    │ 超大规模      │

└──────────────┴────────────┴────────────┴────────────┴──────────────┴──────────────┘
```

## 七、沙箱逃逸与缓解

即使是最强的沙箱也存在逃逸风险，了解威胁才能正确应对。

### 7.1 已知逃逸类型

```
类型 1：内核漏洞利用
  - 描述：利用 Linux 内核 CVE，从容器内提权到宿主机
  - 缓解：定期更新内核；使用 microVM（E2B/Firecracker）彻底隔离内核

类型 2：容器运行时漏洞（runc/containerd）
  - 描述：Docker 运行时本身的 CVE，如 CVE-2019-5736
  - 缓解：定期更新 Docker；使用 rootless 模式；不挂载 docker.sock

类型 3：资源耗尽型 DoS
  - 描述：Fork Bomb、内存耗尽、磁盘写满
  - 缓解：--pids-limit, --memory, --storage-opt

类型 4：侧信道攻击
  - 描述：通过 CPU 缓存时序（如 Spectre）泄露宿主机数据
  - 缓解：hypervisor 级隔离（E2B/Firecracker）

类型 5：供应链污染
  - 描述：沙箱内安装了含恶意代码的 pip 包
  - 缓解：镜像内预装固定版本包，禁用 pip install
```

### 7.2 禁用 pip 安装（镜像加固）

```dockerfile
# Dockerfile 加固版
FROM python:3.11-slim

# 预装所有允许的包
RUN pip install numpy pandas matplotlib scipy

# 删除 pip 本身，防止在容器内安装新包
RUN pip uninstall pip -y
RUN find / -name "pip*" -delete 2>/dev/null || true

# 只读的 site-packages
RUN chmod -R a-w /usr/local/lib/python3.11/site-packages

USER 1000
```

### 7.3 网络隔离验证

```python
# 测试网络是否真的被禁用
def verify_network_isolation(sandbox_run_fn):
    test_code = """
import urllib.request
try:
    urllib.request.urlopen("http://example.com", timeout=3)
    print("FAIL: 网络未被隔离！")
except Exception as e:
    print(f"PASS: 网络已隔离 ({type(e).__name__})")
"""
    result = sandbox_run_fn(test_code)
    assert "PASS" in result, "网络隔离失败！"
    print("网络隔离验证通过")
```

## 八、生产部署 Checklist

```
安全配置
  [ ] 不以 root 运行（--user 1000:1000）
  [ ] 网络完全禁用（--network none）
  [ ] 根文件系统只读（--read-only）
  [ ] 删除所有 Linux 能力（--cap-drop ALL）
  [ ] 内存/CPU/进程数限制已设置
  [ ] seccomp profile 已应用

运营保障
  [ ] 容器超时机制（防止永久占用资源）
  [ ] 容器泄漏监控（定期清理孤儿容器）
  [ ] 执行日志记录（输入代码 + 输出 + 耗时）
  [ ] 异常告警（沙箱逃逸检测）

镜像管理
  [ ] 镜像定期扫描 CVE（Trivy/Snyk）
  [ ] 禁止在容器内安装新包（pip 已删除）
  [ ] 使用 distroless 或最小化基础镜像
  [ ] 镜像版本锁定，禁止 latest 标签

E2B 特有
  [ ] API Key 使用环境变量，不硬编码
  [ ] 设置合理的 timeout（默认 5 分钟）
  [ ] 任务结束后调用 sandbox.close()
  [ ] 监控月度用量，防止账单超支
```

## 本课小结

| 知识点 | 要点 |
|--------|------|
| E2B 原理 | 基于 Firecracker microVM，每次执行独立 VM，~100ms 启动 |
| 状态持久 | E2B 运行在 Jupyter 内核，变量跨调用保留 |
| 文件操作 | `sandbox.files.write/read` 双向传输文件 |
| 选型依据 | 外部用户 → E2B；内部工具 → Docker；低风险 → subprocess |
| 逃逸防御 | 定期更新内核；优先用 VM 级隔离；删除 pip；禁用网络 |
| 生产保障 | 超时 + 泄漏监控 + CVE 扫描 + 执行日志 |
