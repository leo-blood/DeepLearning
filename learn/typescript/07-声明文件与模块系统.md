# 07 声明文件与模块系统

> 声明文件（`.d.ts`）告诉 TypeScript "这里有些东西，相信我"。

---

## 1. 什么是声明文件

当你使用没有 TypeScript 支持的 JavaScript 库，或需要在全局注入某些变量时，需要声明文件：

```ts
// 声明文件不包含实现，只包含类型
declare const VERSION: string;
declare function log(msg: string): void;
declare class EventEmitter { ... }
```

**项目中的例子** — `src/typings/index.d.ts`：

```ts
// 告诉 TS：.png 文件 import 进来是 string（URL）
declare module '*.png' {
  const url: string;
  export = url;
}

declare module '*.jpg' {
  const url: string;
  export = url;
}

declare module '*.svg' {
  const url: string;
  export = url;
}

// 全局变量：由构建工具在运行时注入
declare const envConfig: {
  apiOrigin: {
    user: string;
    training: string;
    schedule: string;
    performance: string;
    trainingRegister: string;
  };
};

// 扩展全局 Window 接口，添加第三方 SDK
interface Window {
  KNB: any;
}
```

## 2. 模块声明 `declare module`

### 2.1 为非 TS 包添加类型

```ts
// 如果某个包没有 @types/xxx，可以自己声明
declare module 'some-legacy-lib' {
  export function doSomething(input: string): void;
  export const VERSION: string;
}
```

### 2.2 为文件类型添加类型（项目中实际用到）

```ts
// 让 TS 知道 import logo from './logo.png' 是合法的
declare module '*.png' {
  const url: string;
  export = url;  // CommonJS 默认导出写法
}

// 等价的 ES Module 写法
declare module '*.png' {
  const url: string;
  export default url;
}
```

### 2.3 为 Vue 文件添加类型

```ts
// src/typings/shims-vue.d.ts（项目中已有）
declare module '*.vue' {
  import type { DefineComponent } from 'vue';
  const component: DefineComponent<{}, {}, any>;
  export default component;
}
```

## 3. 全局变量声明

**项目中的典型场景** — `envConfig`：

```ts
// tsconfig.json 中 "types": ["node", "webpack-env"] 让 TS 知道 process.env
// 但 envConfig 是 webpack.DefinePlugin 注入的自定义全局变量
// 不声明的话 TS 会报"找不到名称 envConfig"

declare const envConfig: {
  apiOrigin: {
    user: string;
    training: string;
  };
};

// 使用时
const userApiUrl = envConfig.apiOrigin.user;  // ✅ 有类型提示
```

## 4. 接口合并（Declaration Merging）

TypeScript 允许多处声明同名接口，最终合并为一个：

```ts
// lib.dom.d.ts（TS 标准库）
interface Window {
  innerWidth: number;
  innerHeight: number;
}

// 你的 src/typings/index.d.ts
interface Window {
  KNB: any;  // 新增字段，合并到 Window
}

// 最终 Window = { innerWidth, innerHeight, KNB, ...所有原生属性 }
window.KNB.openPage({ url: '...' });  // ✅ 不再报错
```

这是扩展第三方类型的标准方式，**不需要修改原始类型文件**。

## 5. 全局类型（不需要 import）

**项目中的 `src/typings/api.d.ts`** 里的类型（像 `MyLearningTaskVo`、`CloudStaffEarningsVO`）
可以在项目所有文件中直接使用，不需要 `import`：

```ts
// src/hooks/myCourse.ts — 没有 import MyLearningTaskVo
const goStudy = ({ studyUrl }: MyLearningTaskVo): void => { ... };
```

**为什么能这样用？**

在 `tsconfig.json` 的 `include` 配置中包含了 `src/**/*.d.ts`，TS 会自动加载所有声明文件中的**全局类型**（即没有 `export` 的声明）。

---

## 6. 模块化 vs 全局

```ts
// 全局声明文件（没有 import/export）
// 文件中的所有 interface/type/declare 都是全局的
declare const API_URL: string;
interface IGlobalUser { ... }

// 模块声明文件（有 import 或 export）
// 变成了普通模块，需要 import 才能使用
export interface IModuleUser { ... }
import type { Something } from 'somewhere';
declare const API_URL: string;  // 加了 import，这里也需要 declare
```

**最佳实践**：
- 真正全局的东西（第三方注入、全局 API 扩展）→ 全局声明文件
- 项目业务类型 → 普通 `.ts` 文件，正常 `export/import`

## 7. `tsconfig.json` 关键配置

```json
{
  "compilerOptions": {
    "strict": true,           // 开启所有严格检查（推荐）
    "noImplicitAny": true,    // 禁止隐式 any（包含在 strict 中）
    "strictNullChecks": true, // null/undefined 必须显式处理（包含在 strict 中）
    "moduleResolution": "node", // 模块解析策略
    "baseUrl": ".",           // 路径别名基础目录
    "paths": {
      "@/*": ["src/*"]        // @ 别名指向 src/
    },
    "types": ["node", "webpack-env"]  // 加载哪些 @types 包的声明
  }
}
```

**项目中 `strict: true` 的影响**：

```ts
// strictNullChecks 下，undefined 不能直接赋给 string
const name: string = undefined;  // ❌

// 必须处理可能为 null/undefined 的情况
const name = user?.profile?.name ?? '未知用户';  // ✅
```

## 8. 路径别名 `@/`

**`tsconfig.json` 中的配置**：

```json
"paths": { "@/*": ["src/*"] }
```

**实际使用**（项目所有文件都在用）：

```ts
// 不用 ../../../api/...，直接用 @/
import { postStudentCenterGetMyLearningTask } from '@/api/types/Learning/api';
import { getStore, IS_DX, setStore } from '@/lib/util';
```

注意：这只是 **类型层面** 的别名。打包时还需要 webpack/vite 的 `alias` 配置配合。

## 9. 实战：看懂 API 层的类型生成

**项目 API 文件结构**：

```
src/api/types/
  Learning/
    api.ts       ← 请求函数（自动生成）
    index.ts     ← 导出汇总
    init.ts      ← 初始化 axios 实例
    types.d.ts   ← 请求/响应类型（自动生成）
```

`types.d.ts` 中声明了所有接口类型，`api.ts` 中的函数使用这些类型，二者配合提供完整的类型安全：

```ts
// types.d.ts（示意）
interface MyLearningTaskVo {
  name: string;
  studyUrl?: string;
  startTime: number;
  endTime: number;
  status: LearningStatus;
  skillList?: SkillVo[];
}

// api.ts（示意）
export async function postStudentCenterGetMyLearningTask(
  params: { taskName?: string }
): Promise<ApiResponse<{ processingLearningTaskList: MyLearningTaskVo[]; ... }>> {
  return axios.post('/student-center/getMyLearningTask', params);
}
```

## 练习

在项目中 `envConfig` 只声明了 5 个 `apiOrigin` 字段。假设新增了一个 `notification` 服务：

```ts
// 如何安全地扩展 envConfig 的类型，而不修改原有声明？
```

<details>
<summary>参考答案</summary>

在 `src/typings/index.d.ts` 中直接修改 `envConfig` 的声明（因为 `declare const` 不支持合并，需要直接更新）：

```ts
declare const envConfig: {
  apiOrigin: {
    user: string;
    training: string;
    schedule: string;
    performance: string;
    trainingRegister: string;
    notification: string;  // 新增
  };
};
```

如果想保持原声明不变（比如它是第三方包提供的），可以用 `declare module` 来增强，但对于 `const` 变量，直接修改 `.d.ts` 文件是最实用的方式。

</details>
