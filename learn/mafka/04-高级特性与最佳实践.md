## 1. bgNameSpace 深度解析

### 1.1 bgNameSpace 是什么

bgNameSpace（业务命名空间）是 Mafka 对 Kafka 集群的逻辑划分，不同 namespace 的 Topic 物理上可能在不同的 Kafka 集群：

```
美团 Mafka 集群分区：
  pingtai        ← 平台通用集群，本项目大多数 Topic 在这里
  xm             ← 大象体系，员工生命周期 Topic 在这里
  com.sankuai.mafka.castle.mtpingtai  ← 特殊业务集群（智管家）
```

### 1.2 本项目的 bgNameSpace 陷阱

```properties
# 绝大多数 Consumer 用 pingtai
mdp.mafka.consumer[0].bgNameSpace=pingtai   # 培训进度
mdp.mafka.consumer[1].bgNameSpace=pingtai   # 激励任务通知
...

# ⚠️ 陷阱1：员工生命周期 Topic 用 xm（大象体系）
mdp.mafka.consumer[8].bgNameSpace=xm         # 不是 pingtai！
mdp.mafka.consumer[8].topicName=staff_life_cycle_handler_topic

# ⚠️ 陷阱2：智管家结果 Topic 用特殊命名空间
mdp.mafka.producer[5].bgNameSpace=com.sankuai.mafka.castle.mtpingtai
mdp.mafka.producer[5].topicName=smart_butler_control_result_topic
```

**后果**：如果 Producer 用 `pingtai`，Consumer 用 `xm`，消息永远收不到，因为消息发到了不同的集群。

## 2. 自产自消模式详解

### 2.1 本项目两个自产自消 Topic

```
Topic: incentive_task_notify_producer
  Producer[0] (appkey=ccsadmin) → 发送激励任务通知
  Consumer[1] (subscribeGroup=incentive_task_notify_consumer) → 接收并处理

Topic: ccsadmin_delay_issue_tasks
  Producer[1] (delay=true) → 发送延迟圈人消息
  Consumer[2] (subscribeGroup=issue_scheduled_incentive_task_consumer) → 到期后接收处理
```

### 2.2 自产自消的死循环风险

```java
// ❌ 危险：Consumer 里又向同一 Topic 发消息

@Service("incentiveTaskNotifyConsumer")
public class IncentiveTaskNotifyConsumer {
    @Autowired
    @Qualifier("incentivesNotifyProducer")
    private IProducerProcessor producer;

    @MdpMafkaMsgReceive
    public ConsumeStatus consume(String msgBody, ...) {
        processNotify(msgBody);
        producer.sendMessage(msgBody);  // ← 又往同一 Topic 发！→ 无限循环！
        return ConsumeStatus.CONSUME_SUCCESS;
    }
}
```

### 2.3 安全的自产自消：用版本号打破循环

```java
// ✅ 安全：用版本号判断是否需要处理
@MdpMafkaMsgReceive
public ConsumeStatus consume(String msgBody, ...) {
    IncentiveDistributeTaskInfoBO bo = JSON.parseObject(msgBody, ...);

    IncentiveTask task = taskMapper.queryById(bo.getTaskId());

    // 版本号校验：如果任务版本已更新，说明有新消息，跳过这条
    if (!bo.getTaskVersion().equals(task.getVersion())) {
        log.info("版本不匹配，跳过, taskId:{}", bo.getTaskId());
        return ConsumeStatus.CONSUME_SUCCESS;
    }

    // 正常处理（不会再发相同 topic 的消息）
    doProcess(task);
    return ConsumeStatus.CONSUME_SUCCESS;
}
```

## 3. 消息积压处理

### 3.1 积压的原因

```
正常流：Producer 每秒发100条 → Consumer 每秒消费100条 → 积压=0

积压场景：
  Producer 每秒发100条
  Consumer 每秒消费30条（慢！）
  → 每秒积压70条
  → 持续1小时 → 积压 = 70 × 3600 = 252,000 条
```

**本项目高风险场景**：员工生命周期 Topic 收到大量入离职事件（如年底）→ 积压。

### 3.2 消费者加速方案

```java
// 方案1：提升消费并发度（mafka.properties 配置）
// 增加消费线程数（与 Kafka 分区数匹配）
mdp.mafka.consumer[8].consumeThreadNums=8   // 默认可能是1

// 方案2：优化单条消费速度
@MdpMafkaMsgReceive
public ConsumeStatus consume(String msgBody, ...) {
    // ❌ 消费中做复杂查询（慢）
    StaffDTO staff = staffPigeonService.getStaffByMis(mis);  // 每条消息都发 RPC

    // ✅ 批量查询，减少 RPC 次数（快）
    // （积压时通常需要专项优化）
    return ConsumeStatus.CONSUME_SUCCESS;
}

// 方案3：消费开关（临时停止消费）
// Lion 配置开关，消费前检查：如果开关关闭，跳过消费，等积压处理完再打开
@MdpConfig("consumer.enable:true")
private Boolean consumerEnable = true;

@MdpMafkaMsgReceive
public ConsumeStatus consume(String msgBody, ...) {
    if (!Boolean.TRUE.equals(consumerEnable)) {
        log.info("消费开关已关闭，消息暂存");
        return ConsumeStatus.CONSUME_FAILURE;  // 不消费，等开关打开
    }
    // 正常消费
}
```

## 5. 消息格式版本管理

### 5.1 消息格式演进问题

```
V1 消息格式（旧）：
  {"taskId":123, "taskVersion":1}

V2 消息格式（新，加了字段）：
  {"taskId":123, "taskVersion":1, "triggerType":"MANUAL"}

问题：
  Producer 升级到 V2，发出 V2 格式消息
  Consumer 还是老版本，解析时 triggerType 字段直接忽略（没问题）

  Producer 降级回 V1（发布回滚），发出 V1 格式消息
  Consumer 已升级到 V2，triggerType 为 null（要做 null 处理）
```

### 5.2 向后兼容的消息格式

```java
// Consumer 的防御性解析
@MdpMafkaMsgReceive
public ConsumeStatus consume(String msgBody, ...) {
    JSONObject msg = JSON.parseObject(msgBody);

    // ✅ 安全获取（有则用，无则默认值）
    Long taskId = msg.getLong("taskId");
    Integer version = msg.getInteger("taskVersion");
    String triggerType = msg.getString("triggerType");   // 可能为 null（旧消息没有）

    if (triggerType == null) {
        triggerType = "AUTO";   // 旧消息默认为自动触发
    }

    // ❌ 不安全（强转可能 NPE）
    // String triggerType = (String) msg.get("triggerType");
}
```

## 6. 监控与排查

### 6.1 消费延迟监控

```
Mafka 控制台可以看到：
  消费延迟（消息发出到被消费的时间差）
  积压量（Topic 中未消费的消息数量）
  消费速率（每秒消费多少条）

告警规则：
  积压量 > 1000 条 → 发大象告警
  消费延迟 > 5分钟 → 发大象告警
```

### 6.2 日志追踪

```java
// Producer 端：记录消息 ID
ProducerResult result = producer.sendMessage(msgBody);
log.info("消息发送成功, taskId:{}, msgId:{}", taskId, result.getMsgId());

// Consumer 端：记录消息 ID
@MdpMafkaMsgReceive
public ConsumeStatus consume(String msgBody, AbstractMdpListener.MdpMqContext ctx) {
    log.info("收到消息, msgId:{}, sendTime:{}, msgBody:{}",
        ctx.getMsgId(), ctx.getSendTime(), msgBody);
    // msgId 用于关联 Producer 和 Consumer 日志
}
```

### 6.3 常见问题排查

```
问题：Consumer 收不到消息
  检查1：bgNameSpace 是否与 Producer 一致
  检查2：topicName 是否拼写正确
  检查3：subscribeGroup 是否是新的（已有 offset 的 group 从上次位置消费）
  检查4：listenerId 是否与 Bean name 匹配
  检查5：Consumer 所在机器是否正常运行

问题：消息重复消费
  原因：Consumer 返回 FAILURE 导致重试
  排查：检查消费逻辑是否有异常，检查是否做了幂等处理

问题：消息顺序错乱
  原因：Mafka 分区内有序，但多分区间无序
  解决：同一业务实体（如同一 taskId）指定相同分区
        producer.sendMessage(msgBody, partitionKey);  // partitionKey=taskId
```
