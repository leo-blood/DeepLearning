## 1. 为什么需要消息队列

### 1.1 同步调用的问题

**场景**：激励任务创建后，需要立即通知 500 个员工。

同步方式：

```java
// 创建任务接口（HTTP，用户在等）
public Result createTask(TaskCreateRequest req) {
    // 1. 保存任务 - 5ms
    taskMapper.insert(task);
    // 2. 给500个员工发大象消息 - 每个10ms = 5000ms !!!
    for (String mis : staffList) {
        xmService.sendMessage(mis, "您有新的激励任务...");
    }
    return Result.success();  // 用户等了5秒，超时了！
}
```

**三大痛点**：
- **慢**：5000ms 响应，用户体验差，HTTP 超时
- **耦合**：任务创建和消息通知绑定在一起，通知系统故障会导致任务创建失败
- **脆**：通知失败会导致整个任务创建回滚

### 1.2 消息队列的解法

```java
// 创建任务接口（解耦后）
public Result createTask(TaskCreateRequest req) {
    // 1. 保存任务 - 5ms
    taskMapper.insert(task);
    // 2. 投递消息 - 1ms（只发消息，不等处理完）
    producer.sendMessage(JSON.toJSONString(task));
    return Result.success();  // 6ms 返回，用户满意
    // 消息队列里的消息，由 Consumer 异步消费，慢慢处理
}
```

### 1.3 消息队列三大价值

```
异步（Async）
  生产者只管发消息，不等消费者处理完
  接口响应时间：5000ms → 6ms

解耦（Decouple）
  生产者不知道消费者是谁，消费者增减不影响生产者
  通知逻辑单独改，不影响任务创建逻辑

削峰（Peak Shaving）
  秒杀：10万请求/秒 → 消息队列缓冲 → Consumer 按 1000/秒 消费
  消费者不被瞬时流量打垮
```

## 2. Mafka 是什么

### 2.1 Mafka 定义

Mafka 是美团基于 Kafka 封装的消息队列，集成在 MDP 中。对开发者来说：

```
Mafka = Kafka 的能力 + 美团基础设施的接入简化

开发者不需要关心：
  - Broker 地址（通过 bgNameSpace 自动发现）
  - Offset 管理（框架自动提交）
  - ZooKeeper 配置（MDP 托管）
  - 监控接入（自动上报 CAT）

开发者只需要关心：
  - 发什么消息（Producer）
  - 收到消息做什么（Consumer）
  - 配置 mafka.properties
```

### 2.2 Mafka 与 Kafka 的对应关系

| Kafka 概念 | Mafka 对应 | 说明 |
|------------|-----------|------|
| Cluster | bgNameSpace | 按业务线隔离的集群命名空间 |
| Topic | topicName | 消息的逻辑分类 |
| Producer | IProducerProcessor | 发消息的 Bean |
| Consumer Group | subscribeGroup | 消费者组，组内竞争消费 |
| Consumer | listenerId 对应的 Bean | 消费消息的 Bean |
| Offset | 框架自动管理 | 记录消费位置 |

## 3. 核心概念详解

### 3.1 Topic（主题）

Topic 是消息的逻辑分类，类似于"频道"：

```
本项目的 Topic 列表（mafka.properties）：

生产的：
  incentive_task_notify_producer   激励任务通知
  ccsadmin_delay_issue_tasks       延迟圈人任务
  ccs_org_process_topic            组织架构处理
  send_wechat_message_topic        发微信消息
  trigger_message_delivery_task_topic  消息投递触发
  smart_butler_control_result_topic    智能管家控制结果

消费的（12个，含上面的自产自消）
```

**命名规范**：`{业务域}_{动作}_{类型}`，如 `incentive_task_notify_producer`。

### 3.2 bgNameSpace（业务命名空间）

bgNameSpace 是 Mafka 特有概念，决定消息走哪个物理集群：

```properties
# 绝大多数 Topic 用 pingtai（平台集群）
mdp.mafka.producer[0].bgNameSpace=pingtai

# 智能管家 Topic 用特殊集群（castle.mtpingtai）
mdp.mafka.producer[5].bgNameSpace=com.sankuai.mafka.castle.mtpingtai

# 员工生命周期 Topic 来自大象体系（xm）
mdp.mafka.consumer[8].bgNameSpace=xm
```

**重要**：Producer 和 Consumer 的 bgNameSpace 必须一致，否则收不到消息。这是本项目的一个踩坑点（`staff_life_cycle_handler_topic` 的 bgNameSpace 是 `xm`，与其他 Topic 不同）。

### 3.3 ConsumerGroup（消费者组）

```
同一个 Topic 有 3 台机器消费：

subscribeGroup = "incentive_task_notify_consumer_group"
  机器A  ←── 消息1
  机器B  ←── 消息2     ← 组内每条消息只被一台机器消费（竞争）
  机器C  ←── 消息3

不同 subscribeGroup 的消费者都能收到所有消息：
  Group_A: 机器A消费消息1，机器B消费消息2...
  Group_B: 机器X也消费消息1，机器Y也消费消息2...（广播效果）
```

本项目每个 Topic 只有一个消费者组。

### 3.4 消息投递保证（at-least-once）

Mafka 保证消息**至少投递一次**（at-least-once），不保证恰好一次：

```
正常情况：
  Producer 发消息 → Consumer 消费 → 提交 offset ✅

异常情况（Consumer 消费完但提交 offset 前崩溃）：
  Producer 发消息 → Consumer 消费 → 崩溃
                                   ↓
                           Mafka 认为未消费 → 重新投递
                                   ↓
                           Consumer 重新消费（重复！）
```

**因此所有消费者逻辑必须幂等**（相同消息处理多次，结果不变）。

## 4. 消息格式设计

### 4.1 本项目的消息格式（JSON 字符串）

所有消息体都是 JSON 字符串，包含足够信息：

```java
// 延迟圈人消息格式
// IncentiveDistributeTaskInfoBO
{
    "taskId": 12345,         // 任务ID（用于查询完整信息）
    "taskVersion": 3         // 版本号（用于幂等校验）
}

// 培训进度通知消息格式
{
    "staffMis": "zhangsan",
    "staffName": "张三",
    "status": 2,                  // 2=已完成
    "completeTime": 1716977061757,
    "contentCollectionId": 3175,  // 内容包ID
    "taskBatchId": 2930           // 指派任务ID
}
```

### 4.2 消息格式设计原则

```
✅ 包含版本/状态用于幂等校验
✅ 包含业务主键，Consumer 可以查到完整数据
✅ 包含时间戳，用于时序校验
✅ 字段语义清晰，有注释

❌ 只传 ID，Consumer 还得查库
❌ 传大对象（Consumer 只用几个字段）
❌ 没有版本控制，格式改了Consumer直接报错
```

## 5. mafka.properties 配置结构速览

```properties
# ===== 生产者（Producer）配置 =====
# 索引从 [0] 开始

mdp.mafka.producer[0].producerName=incentivesNotifyProducer   # Spring Bean 名
mdp.mafka.producer[0].bgNameSpace=pingtai                      # 集群命名空间
mdp.mafka.producer[0].appkey=com.sankuai.csccratos.ccsadmin   # 应用标识
mdp.mafka.producer[0].topicName=incentive_task_notify_producer # Topic
mdp.mafka.producer[0].delay=true                               # 支持延迟消息

# ===== 消费者（Consumer）配置 =====
# 索引从 [0] 开始

mdp.mafka.consumer[0].bgNameSpace=pingtai
mdp.mafka.consumer[0].appkey=com.sankuai.csccratos.ccsadmin
mdp.mafka.consumer[0].topicName=cloud.customer.service.improve # 订阅的 Topic
mdp.mafka.consumer[0].subscribeGroup=training.progress.notify.consumer.group
mdp.mafka.consumer[0].listenerId=trainingProgressNotifyConsumer  # 对应的 Bean 名
```
