# Mafka 消费者开发

> 本文聚焦：如何配置 Consumer，实现可靠的消息消费。

## 1. Consumer 配置详解

### 1.1 mafka.properties Consumer 配置

```properties
# Consumer 配置项说明
mdp.mafka.consumer[0].bgNameSpace=pingtai
#   消息集群命名空间，必须与 Producer 端的 bgNameSpace 完全一致

mdp.mafka.consumer[0].appkey=com.sankuai.csccratos.ccsadmin
#   消费者的应用标识

mdp.mafka.consumer[0].topicName=cloud.customer.service.improve
#   订阅的 Topic 名称

mdp.mafka.consumer[0].subscribeGroup=training.progress.notify.consumer.group
#   消费者组名：同一组内的多台机器竞争消费，每条消息只被消费一次

mdp.mafka.consumer[0].listenerId=trainingProgressNotifyConsumer
#   对应的 Spring Bean 名称（@Service 的 Bean name）
```

### 1.2 listenerId 与 Bean name 的对应

```java
// listenerId = "trainingProgressNotifyConsumer"

// 方式1：类名首字母小写（默认 Bean name）
@Service     // Bean name = "trainingProgressNotifyConsumer"（自动推导）
public class TrainingProgressNotifyConsumer { ... }

// 方式2：显式指定 Bean name
@Service("trainingResultNotifyConsumer")  // 显式指定
public class TrainingResultNotifyConsumer { ... }
// 对应配置：listenerId=trainingResultNotifyConsumer

// ❌ 不一致时启动报错
// listenerId=trainingProgressNotify（缺少Consumer）
// Bean name=trainingProgressNotifyConsumer
// → 找不到对应的 Bean
```

## 2. 消费方法编写

### 2.1 标准消费方法签名

```java
@Service
@Slf4j
public class TrainingProgressNotifyConsumer {

    @MdpMafkaMsgReceive   // ← 标记这是消费方法，框架自动调用
    public ConsumeStatus exec(
        String msgBody,                            // 消息体（JSON 字符串）
        AbstractMdpListener.MdpMqContext context   // 消息上下文
    ) {
        // 消费逻辑
        return ConsumeStatus.CONSUME_SUCCESS;      // 返回消费结果
    }
}
```

### 2.2 ConsumeStatus 含义

| 返回值 | 含义 | 框架行为 |
|--------|------|---------|
| `CONSUME_SUCCESS` | 消费成功 | 提交 offset，不重试 |
| `CONSUME_FAILURE` | 消费失败 | 不提交 offset，等待重试 |

**重要决策**：何时返回 FAILURE？

```java
// ✅ 返回 SUCCESS 的场景：
// 1. 正常处理完毕
return ConsumeStatus.CONSUME_SUCCESS;

// 2. 业务异常（如数据不符合条件）：不需要重试，跳过
} catch (BusiException e) {
    log.info("业务校验不通过，跳过消息: {}", msgBody, e);
    return ConsumeStatus.CONSUME_SUCCESS;  // 跳过，不重试

// ✅ 返回 FAILURE 的场景：
// 3. 系统异常（如 DB 故障）：需要重试，等恢复
} catch (Exception e) {
    log.error("消费异常，等待重试: {}", msgBody, e);
    return ConsumeStatus.CONSUME_FAILURE;  // 重试
```

### 2.3 完整的消费方法模板

```java
@Service
@Slf4j
public class TrainingProgressNotifyConsumer {

    @Autowired
    private IncentiveMapperService incentiveMapperService;

    @Autowired
    private IncentiveCommonService commonService;

    @MdpMafkaMsgReceive
    public ConsumeStatus exec(String msgBody, AbstractMdpListener.MdpMqContext ctx) {
        long startTime = System.currentTimeMillis();
        log.info("TrainingProgressNotifyConsumer 收到消息: {}", msgBody);

        try {
            // 1. 解析消息
            TrainingProgressBO bo = parseMessage(msgBody);

            // 2. 业务处理
            consume(bo);

            log.info("TrainingProgressNotifyConsumer 消费完成, 耗时:{}ms",
                System.currentTimeMillis() - startTime);
            return ConsumeStatus.CONSUME_SUCCESS;

        } catch (BusiException e) {
            // 业务异常：不需要重试
            log.info("TrainingProgressNotifyConsumer 业务校验不通过, msg:{}, error:{}",
                msgBody, e.getMessage());
            return ConsumeStatus.CONSUME_SUCCESS;   // ← 返回成功，跳过该消息

        } catch (Exception e) {
            // 系统异常：需要重试
            log.error("{}, TrainingProgressNotifyConsumer 系统异常, msg:{}",
                CommonConstant.LOG_PREFIX_CLOUD_CUSTOMER_SERVICE_PLATFORM, msgBody, e);
            return ConsumeStatus.CONSUME_FAILURE;   // ← 返回失败，等待重试
        }
    }

    private TrainingProgressBO parseMessage(String msgBody) {
        // 解析 JSON，字段缺失时用默认值
        return JSON.parseObject(msgBody, TrainingProgressBO.class);
    }
}
```

## 3. MdpMqContext 的使用

```java
@MdpMafkaMsgReceive
protected ConsumeStatus receive(String msgBody, AbstractMdpListener.MdpMqContext ctx) {
    // context 提供消息的元数据

    // 消息 ID（用于追踪）
    String msgId = ctx.getMsgId();

    // 消息的发送时间
    long sendTime = ctx.getSendTime();

    // 当前消费已重试次数
    int retryCount = ctx.getRetryCount();
    if (retryCount >= 3) {
        // 已重试3次仍失败，记录到数据库，人工处理
        failedMessageService.save(msgBody, "重试超限");
        return ConsumeStatus.CONSUME_SUCCESS;   // 不再重试
    }

    // Topic 名
    String topic = ctx.getTopic();

    // ... 业务逻辑
}
```

## 4. 幂等性设计（重点！）

### 4.1 为什么必须幂等

```
Mafka 的保证：at-least-once（至少一次）

场景：
  Consumer 处理完消息，提交 offset 前 → JVM 崩溃
  Mafka 认为消息未被消费 → 重新投递
  同一条消息被处理两次！
```

### 4.2 幂等方案一：版本号校验

```java
// 适用于：激励任务延迟圈人
@MdpMafkaMsgReceive
public ConsumeStatus consume(String msgBody, ...) {
    IncentiveDistributeTaskInfoBO bo = JSON.parseObject(msgBody, ...);

    // 查当前版本
    IncentiveTask task = incentiveMapperService.queryById(bo.getTaskId());

    // 版本不匹配：任务已被修改，这条消息过时了，跳过
    if (!bo.getTaskVersion().equals(task.getVersion())) {
        log.info("版本不匹配，跳过, taskId:{}, msgVer:{}, dbVer:{}",
            bo.getTaskId(), bo.getTaskVersion(), task.getVersion());
        return ConsumeStatus.CONSUME_SUCCESS;
    }

    // 版本匹配，正常处理
    doProcess(task);
    return ConsumeStatus.CONSUME_SUCCESS;
}
```

### 4.3 幂等方案二：唯一索引

```java
// 适用于：培训进度更新（每个员工+任务只更新一次）
@MdpMafkaMsgReceive
public ConsumeStatus consume(String msgBody, ...) {
    TrainingProgressBO bo = parseMessage(msgBody);

    try {
        // 插入时数据库唯一索引检查（staffMis + taskBatchId 唯一）
        incentiveStaffMapper.insertOrUpdate(bo.getStaffMis(), bo.getTaskBatchId(), bo.getStatus());
    } catch (DuplicateKeyException e) {
        // 唯一索引冲突 = 重复消费，忽略
        log.info("重复消息，已忽略, staffMis:{}, taskId:{}", bo.getStaffMis(), bo.getTaskBatchId());
    }

    return ConsumeStatus.CONSUME_SUCCESS;
}
```

### 4.4 幂等方案三：状态机校验

```java
// 适用于：只有特定状态才处理
@MdpMafkaMsgReceive
public ConsumeStatus consume(String msgBody, ...) {
    IncentiveTask task = incentiveMapperService.queryById(taskId);

    // 只有"待下发"状态才处理，其他状态说明已经处理过了
    if (task.getStatus() != TaskStatusEnum.TO_BE_ISSUED.getCode()) {
        log.info("任务状态已变更，跳过, taskId:{}, status:{}", taskId, task.getStatus());
        return ConsumeStatus.CONSUME_SUCCESS;
    }

    // 执行下发逻辑
    doIssue(task);
    return ConsumeStatus.CONSUME_SUCCESS;
}
```

## 6. 消费重试与死信处理

### 6.1 重试机制

```
CONSUME_FAILURE 被返回后：

重试策略（Mafka 默认）：
  第1次失败 → 等待几秒后重试
  第2次失败 → 等待更长时间重试
  第3次失败 → ...
  超过最大重试次数 → 消息进入死信队列（或丢弃）

重试期间，消费者组的其他消息可能被阻塞（取决于分区）
→ 所以非必要不要返回 CONSUME_FAILURE！
```

### 6.2 人工处理死信

```java
// 超过重试次数的消息，记录到数据库，人工处理
@MdpMafkaMsgReceive
protected ConsumeStatus receive(String msgBody, AbstractMdpListener.MdpMqContext ctx) {
    int retryCount = ctx.getRetryCount();
    try {
        process(msgBody);
        return ConsumeStatus.CONSUME_SUCCESS;
    } catch (Exception e) {
        if (retryCount >= 3) {
            // 记录到人工处理队列
            log.error("消息重试超限，已记录人工处理, msgBody:{}", msgBody, e);
            failedMessageRepository.save(new FailedMessage(msgBody, e.getMessage()));
            return ConsumeStatus.CONSUME_SUCCESS;   // 停止重试
        }
        log.error("消费失败，准备重试 (retry:{}/3), msgBody:{}", retryCount, msgBody, e);
        return ConsumeStatus.CONSUME_FAILURE;
    }
}
```
