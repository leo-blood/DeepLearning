## 1. Cron 表达式完整指南

### 1.1 Quartz Cron 格式（6位）

```
秒(0-59) 分(0-59) 时(0-23) 日(1-31) 月(1-12) 周(1-7, 1=周日)

特殊字符：
  *  任意值
  ?  不关心（日/周 互斥时用）
  -  范围（如 1-5 = 周一到周五）
  ,  枚举（如 1,3,5 = 周一/三/五）
  /  步长（如 0/30 = 每30分钟）
  L  最后（如 L = 月最后一天）
```

### 1.2 本项目常用 Cron 解析

```
凌晨相关：
  0 0 1 * * ?     每天凌晨1点（ccsStaffLevelJob，RuleReminderJob）
  0 0 2 * * ?     每天凌晨2点（incentiveStatusUpdate）
  0 0 3 * * ?     每天凌晨3点（doIncentiveTask）

夜间相关：
  0 0 23 * * ?    每天23点（taskRewardPenaltyJob）
  0 30 22 * * ?   每天22:30

高频任务：
  0 */5 * * * ?   每5分钟一次
  0 0 */1 * * ?   每小时一次

工作日：
  0 0 9 * * 2-6   工作日（周一到周五）早上9点（注意：1=周日）

月度任务：
  0 0 1 1 * ?     每月1日凌晨1点
```

### 1.3 Cron 验证

```
在配置前验证 Cron 表达式：
  1. 使用在线工具：https://crontab.cronhub.io/
  2. 在控制台"试算"功能：输入 cron，展示未来10次触发时间

常见错误：
  日和周同时设置非?值 → Quartz 报错
  0 0 1 15 * 5 → 每月15日且是周五才触发（非常少见，可能是错误）
  0 0 1 15 * ? → 每月15日凌晨1点（正确）
  0 0 1 ? * 5  → 每周五凌晨1点（正确）
```

## 2. 控制台核心操作

### 2.1 查看任务列表

```
Crane 控制台 → 任务管理 → 选择 AppKey: com.sankuai.csccratos.ccsadmin

展示字段：
  任务名                    注册时间    cron            下次执行时间    状态
  ccsStaffLevelJob         2024-01    0 0 1 * * ?    2024-01-16 01:00  启用
  taskRewardPenaltyJob     2024-01    0 0 23 * * ?   2024-01-15 23:00  启用
  incentiveStatusUpdate    2024-01    0 0 2 * * ?    2024-01-16 02:00  启用
```

### 2.2 手动触发任务

**使用场景**：
- 开发完成后验证逻辑
- 补跑某天失败的数据
- 运营临时要求立即刷新

**操作步骤**：

```
1. 找到目标任务（如 ccsStaffLevelJob）
2. 点击"立即执行"按钮
3. 填写执行参数：
     参数值：2024-01-10（补跑1月10日的数据）
     目标机器：选择一台或默认（任意一台）
4. 确认执行
5. 在"执行历史"中查看结果
```

### 2.3 修改 Cron 表达式

```
场景：线上发现员工等级计算影响早高峰，需要改到凌晨2点

操作：
  1. 找到 ccsStaffLevelJob
  2. 点击"修改配置"
  3. cron 从 "0 0 1 * * ?" 改为 "0 0 2 * * ?"
  4. 保存
  5. 效果：下次触发时间立即变为凌晨2点（不需要重启应用）

注意：修改立即生效，如果原 cron 下次触发是今晚1点，修改后今晚不会触发
```

### 2.4 暂停/恢复任务

```
暂停（临时停止）：
  场景：系统维护期间，暂停所有激励任务相关 Job
  操作：找到任务 → 点击"暂停"→ 任务不再自动触发（手动触发仍可用）

恢复：
  操作：找到已暂停任务 → 点击"启用"→ 恢复按 cron 自动触发
```

## 3. 执行历史分析

### 3.1 查看执行历史

```
执行历史字段：
  执行时间        机器IP         触发方式    参数         耗时      结果
  2024-01-15 01:00  10.0.1.23   定时触发    (空)        4m32s    成功
  2024-01-14 01:00  10.0.1.24   定时触发    (空)        4m18s    成功
  2024-01-13 01:00  10.0.1.23   定时触发    (空)        失败     java.lang.OOM...
  2024-01-13 01:05  10.0.1.24   手动触发    2024-01-13  2m45s    成功（补跑）
```

### 3.2 从历史判断问题

```
耗时分析：
  正常：4分钟
  今天：40分钟 → 数据量增加了？DB 慢查询？
  趋势：每天增加1分钟 → 数据量线性增长，需要优化

失败分析：
  连续2天失败 → 代码有 Bug 或依赖服务故障
  偶发失败 → 可能是网络超时，检查是否幂等可重跑

机器分析：
  每次都是同一台机器执行 → 可能负载不均衡
  某台机器失败率高 → 该机器可能有问题
```

## 4. 本项目高风险 Job 运维

### 4.1 IncentiveTaskJob（最高风险）

```
风险说明：
  激励任务状态流转的核心，影响所有激励任务
  状态一旦跳跃（如从"待下发"直接到"已结束"），数据无法恢复

操作规范：
  - 修改此 Job 前，必须通过完整的状态流转测试用例（docs/tests/cases.md）
  - 手动触发前，先在 staging 环境验证
  - 生产补跑时，先查清楚数据当前状态，避免重复流转

排查命令（排查状态异常）：
  SELECT id, status, task_start_date, task_end_date
  FROM incentive_task
  WHERE status IN (1,2,3,4,5,6)    -- 非终态的任务
  ORDER BY task_start_date;
```

### 4.2 CcsStaffLevelJob（性能敏感）

```
风险说明：
  每日凌晨批量计算所有员工等级，数据量大，性能敏感

监控指标：
  正常耗时：约4-5分钟
  告警阈值：超过30分钟（说明性能问题）

优化手段（通过 Lion 配置，不重启）：
  Lion: default.org.staff.amount = 100  ← 减小每批量，减轻 DB 压力
  Lion: default.org.staff.amount = 500  ← 增大批量，加快执行速度

发生 OOM 时：
  1. 检查 Lion 配置 default.org.staff.amount，调小
  2. 检查 calcAndSaveLevel 中是否有大对象创建
  3. 临时增大 JVM 内存（JVM_EXT_ARGS=-Xmx4g）
```

### 4.3 CloudOfficeStaffDesktopJob（依赖外部 RPC）

```
风险说明：
  依赖华为云桌面 RPC 服务，外部服务不稳定时会失败

处理方案：
  1. 华为云 RPC 超时 → 任务执行失败 → 控制台可以手动重跑
  2. 批量分配失败 → 单个失败不影响其他 → 查日志找到失败的员工 → 手动重跑

重跑参数：
  传入失败员工的 mis 列表（自定义参数格式，由 Job 代码支持）
```

## 5. 告警与应急响应

### 5.1 建议的告警配置

```
Crane 控制台告警 or CAT 监控：

1. 执行失败告警
   条件：任务执行失败
   通知：大象通知 on-call 人员

2. 耗时超时告警
   条件：执行时间 > 预期时间 × 3
   通知：大象通知

3. 未执行告警
   条件：计划执行时间过了30分钟仍未执行
   场景：Crane Server 故障、所有机器宕机
```

### 5.2 紧急情况处理

```
场景：凌晨1点员工等级计算任务失败

Step1：收到告警通知
Step2：查看 Crane 执行历史，确认失败原因（看异常栈）
Step3：判断是否影响今天的业务（等级展示、奖励发放等）
Step4：根据根因处理：
        DB 连接池耗尽 → 重启应用 → 重新手动触发
        代码 Bug → 紧急修复 → 重新手动触发（补跑昨天日期）
        数据问题 → 修复数据 → 重新手动触发
Step5：确认重跑成功，数据正确
Step6：写故障复盘（参考 docs/knowledge/issues.md 格式）
```

## 6. Crane 任务开发检查清单

```
开发新任务前：
  ☑ 确认任务名唯一（与现有 25+ 个任务不重复）
  ☑ 在 AGENTS.md 任务列表中登记
  ☑ 设计幂等方案（重跑数据不会重复）
  ☑ 考虑数据量，设计分批处理

编码时：
  ☑ @Crane 注解值与控制台注册名一致
  ☑ 方法被 try-catch 包裹，不抛异常
  ☑ 有开始/结束日志（含耗时）
  ☑ 大批量数据分页处理
  ☑ 单条失败不中断整体

上线后：
  ☑ 在 Crane 控制台找到新任务（确认注册成功）
  ☑ 手动触发一次，验证逻辑正确
  ☑ 查看执行历史，确认结果
  ☑ 配置 Cron 表达式
  ☑ 配置告警规则
```
