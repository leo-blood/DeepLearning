# Crane 任务设计模式

> 本文聚焦：如何设计幂等、高效、可靠的定时任务。

---

## 1. 幂等性设计

### 1.1 为什么 Crane 任务需要幂等

场景一：手动补跑（正常需求）

```
场景：昨天的员工等级任务跑了但数据有问题
操作：在 Crane 控制台手动触发，传入昨天的日期
结果：希望重跑一遍，但不能让数据重复
```

场景二：网络问题导致重复触发（异常情况）

```
Crane 控制台发指令 → 机器A 收到并执行
                  → 网络抖动，Crane 认为没执行
                  → 重新下发 → 机器A 又执行了一次
```

### 1.2 幂等方案：执行标记

```java
@Crane("ccsStaffLevelJob")
public void calcStaffLevel(String dateStr) {
    // 没传日期，用今天
    String targetDate = StringUtils.isBlank(dateStr) ?
        DateUtils.formatDate(new Date()) : dateStr;

    // 检查是否已经计算过
    StaffLevelJobRecord record = jobRecordMapper.queryByDate(targetDate);
    if (record != null && "SUCCESS".equals(record.getStatus())) {
        log.info("[ccsStaffLevelJob] {} 已计算完成，跳过", targetDate);
        return;
    }

    // 标记开始
    jobRecordMapper.insertOrUpdate(new StaffLevelJobRecord(targetDate, "RUNNING"));

    try {
        doCalcStaffLevel(targetDate);

        // 标记成功
        jobRecordMapper.updateStatus(targetDate, "SUCCESS");
        log.info("[ccsStaffLevelJob] {} 计算完成", targetDate);

    } catch (Exception e) {
        // 标记失败（允许下次重跑）
        jobRecordMapper.updateStatus(targetDate, "FAILED");
        log.error("[ccsStaffLevelJob] {} 计算失败", targetDate, e);
    }
}
```

### 1.3 幂等方案：UPDATE WHERE 状态判断

```java
@Crane("incentiveStatusUpdate")
public void updateTaskStatus(String dateStr) {
    Date now = parseDate(dateStr);

    // 使用 UPDATE WHERE status=? 保证幂等
    // 即使同时执行两次，MySQL UPDATE 的原子性保证只有一个 UPDATE 成功
    int updated = incentiveTaskMapper.updateStatusIfPending(
        TaskStatusEnum.ISSUED.getCode(),   // 新状态
        TaskStatusEnum.TO_BE_ISSUED.getCode(),  // 原状态（条件）
        now                                     // 开始时间 <= now
    );
    log.info("[incentiveStatusUpdate] 状态流转: 待下发→已下发, 更新{}条", updated);
}
```

对应的 SQL：

```sql
UPDATE incentive_task
SET status = #{newStatus}, update_time = NOW()
WHERE status = #{oldStatus}
  AND task_start_date <= #{now}
```

## 2. 分批处理与 OOM 防范

### 2.1 性能敏感的 ccsStaffLevelJob

员工等级计算是本项目最重的 Job，设计要点：

```java
@Crane("ccsStaffLevelJob")
public void calcStaffLevel(String param) {
    int pageSize = lionService.defaultOrgStaffAmount != null ?
        lionService.defaultOrgStaffAmount : 200;   // 从 Lion 动态配置批量大小

    int pageNum = 1;
    int totalProcessed = 0;
    long startTime = System.currentTimeMillis();

    log.info("[ccsStaffLevelJob] 开始, pageSize:{}", pageSize);

    while (true) {
        List<StaffInfo> batch;
        try {
            // 分页查询（避免一次性加载所有数据 → OOM）
            batch = staffMapper.queryByPage(pageNum, pageSize);
        } catch (Exception e) {
            log.error("[ccsStaffLevelJob] 查询第{}页失败", pageNum, e);
            break;
        }

        if (CollectionUtils.isEmpty(batch)) break;

        // 处理这一批（每条独立 try-catch，单条失败不影响整批）
        for (StaffInfo staff : batch) {
            try {
                staffLevelService.calcAndSaveLevel(staff);
                totalProcessed++;
            } catch (Exception e) {
                log.error("[ccsStaffLevelJob] 员工等级计算失败, staffMis:{}", staff.getMis(), e);
            }
        }

        log.info("[ccsStaffLevelJob] 第{}页完成, 本批:{}, 累计:{}, 耗时:{}ms",
            pageNum, batch.size(), totalProcessed, System.currentTimeMillis() - startTime);

        if (batch.size() < pageSize) break;  // 最后一页
        pageNum++;

        // 批次间小憩，避免给 DB 造成过大压力
        ThreadUtils.sleep(50);
    }

    log.info("[ccsStaffLevelJob] 全部完成, 总处理:{}, 总耗时:{}ms",
        totalProcessed, System.currentTimeMillis() - startTime);
}
```

### 2.2 动态批量大小（Lion 配置）

```java
// 批量大小从 Lion 读取，线上可以动态调整
// Lion key: default.org.staff.amount
```

## 3. 激励任务状态流转深析

### 3.1 7个状态的流转逻辑

```
待下发(1) ──任务开始时间到达──▶ 已下发(2)
已下发(2) ──任务结束时间到达──▶ 考核整改中(3)
考核整改中(3) ──整改期结束──▶ 待管控(4)
待管控(4) ──管控期开始──▶ 待处罚(5)
待处罚(5) ──处罚生效──▶ 处罚生效中(6)
处罚生效中(6) ──处罚失效日期到达──▶ 已结束(7)
```

### 3.2 为什么禁止在 HTTP 接口修改状态（ADR-001）

```
如果 HTTP 接口也能改状态：

时间线：
  T=0: Crane Job 查询 → 任务A 是"待下发"，准备改"已下发"
  T=1: HTTP 接口请求 → 也在修改任务A 的状态
  T=2: Crane Job UPDATE → 改成"已下发"
  T=3: HTTP 接口 UPDATE → 也改成了"已下发"（或改成了别的状态！）
  
后果：状态被修改两次，可能跳过某个状态节点，导致业务逻辑混乱。

规则：状态字段只由 Crane Job 修改，HTTP 接口不允许碰 status 字段。
```

### 3.3 状态流转的实现细节

```java
// IncentiveTaskJob.java（核心代码逻辑）

public void updateTaskStatus(String dateStr) {

    int count2 = incentiveTaskMapper.updateToAssessing(now);
    log.info("已下发→考核整改中: {}条", count2);

}
```

## 4. 任务拆分与依赖

### 4.1 何时需要拆分 Job

```
场景：云办公桌面分配，流程复杂，拆成多个 Job：

cloudOfficeStaffDesktopJob        主任务（查询待分配员工）
    ↓ 之后手动/自动触发
cloudOfficeStaffDesktopCreateJob  子任务（创建账号）
    ↓
cloudOfficeStaffDesktopInitJob    子任务（初始化配置）
    ↓
cloudOfficeStaffDesktopRemindJob  子任务（到期提醒）

好处：
  - 每个 Job 职责单一，易于调试
  - 子任务失败只重跑该子任务，不影响主任务
  - 可以独立调整各步骤的执行时间
```

### 4.2 主任务触发子任务的方式

```java
// 方式1：通过 MQ（推荐，解耦）
@Crane("cloudOfficeStaffDesktopJob")
public void mainJob(String param) {
    List<Staff> pendingStaff = getPendingStaff();
    for (Staff staff : pendingStaff) {
        // 把每个员工作为消息发出，由子 Consumer 处理
        producer.sendMessage(JSON.toJSONString(staff));
    }
}

// 方式2：直接调用（适合强耦合的步骤）
@Crane("cloudOfficeStaffDesktopJob")
public void mainJob(String param) {
    List<Staff> pendingStaff = getPendingStaff();
    cloudOfficeStaffDesktopService.distribute(pendingStaff);  // 内部完成所有步骤
}
```

## 5. 补偿与告警

### 5.1 失败补偿设计

```java
// CcsOrgProcessJob.java 的补偿机制参考
@Crane("ccsOrgDailyReconciliationTask")
public void dailyReconciliation(String param) {
    // 1. 执行主流程
    try {
        orgService.reconcile();
    } catch (Exception e) {
        log.error("对账失败", e);

        // 2. 写入补偿表
        compensationMapper.insert(new Compensation(
            "ccsOrgDailyReconciliationTask",
            param,
            e.getMessage(),
            new Date()
        ));
    }

    // 3. 处理之前失败的补偿记录
    List<Compensation> pendingCompensations =
        compensationMapper.queryPending("ccsOrgDailyReconciliationTask");
    for (Compensation comp : pendingCompensations) {
        try {
            orgService.reconcileByDate(comp.getParam());
            compensationMapper.markSuccess(comp.getId());
        } catch (Exception e) {
            log.error("补偿执行失败", e);
            compensationMapper.incrementRetry(comp.getId());
        }
    }
}
```

### 5.2 执行监控

```java
@Crane("ccsStaffLevelJob")
public void calcStaffLevel(String param) {
    long start = System.currentTimeMillis();
    try {
        doCalc();
        long duration = System.currentTimeMillis() - start;

        // 执行超过30分钟，发告警
        if (duration > 30 * 60 * 1000) {
            xmService.sendAlert("ccsStaffLevelJob 执行耗时超30分钟: " + duration + "ms");
        }
    } catch (Exception e) {
        // 执行失败，发告警
        xmService.sendAlert("ccsStaffLevelJob 执行失败: " + e.getMessage());
        log.error("[ccsStaffLevelJob] 执行失败", e);
    }
}
```
