## 一、为什么需要分布式系统？

### 1.1 单机架构的瓶颈

```
单机架构问题：
┌─────────────────────────────┐
│  单台服务器                  │
│  ┌───────┐  ┌───────────┐  │
│  │ 业务  │  │   数据库   │  │
│  └───────┘  └───────────┘  │
│                             │
│  ❌ CPU/内存有上限           │
│  ❌ 宕机 = 全站不可用        │
│  ❌ 无法水平扩展             │
└─────────────────────────────┘
```

### 1.2 分布式架构带来的挑战

| 挑战 | 具体问题 | 解决方向 |
|------|---------|---------|
| **服务间通信** | A 如何调用 B？网络超时怎么办？ | RPC 框架 |
| **异步解耦** | 高峰期请求洪峰如何削峰？ | 消息队列 |
| **任务协调** | 定时任务在多台机器上重复执行？ | 分布式调度 |
| **状态共享** | 多实例如何共享缓存/锁？ | 分布式缓存 |
| **参数管理** | 上线后如何不重启修改配置？ | 配置中心 |

## 三、五大框架一览

### 3.1 Pigeon — RPC 框架

**解决的问题：** 服务 A 如何像调用本地方法一样调用服务 B？

```
没有 RPC：                    有了 Pigeon：
A 服务                        A 服务
  ↓ 手动 HTTP 请求               ↓ @MdpPigeonClient 注入
  ↓ 处理序列化/反序列化           ↓ 直接调用接口方法（透明）
  ↓ 重试、超时、服务发现          ↓ 框架自动处理一切
B 服务                        B 服务
```

**核心特性：**
- 自动服务注册与发现（MNS）
- Hessian 序列化（比 JSON 更紧凑）
- 内置超时/重试/熔断
- 全链路追踪

### 3.2 Mafka — 消息队列

**解决的问题：** 如何实现服务间异步通信、削峰填谷？

```
同步调用问题：                异步消息方案：
A → B → C → D               A → [Mafka Topic]
  每步都要等待                     ↓        ↓
  任何一步慢都阻塞           B消费      C消费
                           (并行、解耦、缓冲)
```

**核心特性：**
- 基于 Kafka，天然支持高吞吐
- 支持延迟消息（N秒后投递）
- 消费失败自动重试
- 至少一次投递语义

### 3.3 Crane — 分布式定时任务

**解决的问题：** 多实例部署时如何保证定时任务只执行一次？

```
@Scheduled 的问题：          Crane 的方案：
3台机器同时跑定时任务          分布式锁 → 只有1台执行
→ 重复处理数据                控制台可视化
→ 数据错乱                    手动触发/暂停/查历史
```

### 3.4 Squirrel — 分布式缓存与锁

**解决的问题：** 多实例共享 Redis，如何避免 Key 冲突和竞态条件？

```
原始 Redis 问题：             Squirrel 的方案：
所有服务共用一个 Redis         StoreKey = 命名空间隔离
Key 可能冲突                   类型安全（编译期检查）
分布式锁手写容易出错           内置分布式锁 API
```

### 3.5 Lion — 配置中心

**解决的问题：** 如何在不重启服务的情况下修改配置参数？

```
配置演进史：
代码硬编码 → 配置文件 → 环境变量 → Lion 配置中心
                                    ↓
                              修改后实时推送所有实例
                              支持环境隔离（dev/staging/prod）
                              支持 KMS 加密敏感配置
```

## 四、五大框架选型指南

| 场景 | 应使用的框架 | 原因 |
|------|------------|------|
| 需要调用另一个服务的接口 | Pigeon | 同步 RPC，有返回值 |
| 通知其他服务"发生了某事" | Mafka | 异步，发送方不等待 |
| 每天凌晨跑一次数据同步 | Crane | 分布式调度，不重复执行 |
| 缓存热点数据 | Squirrel | 封装 Redis，命名空间隔离 |
| 多服务竞争同一资源 | Squirrel | 分布式锁 |
| 功能开关/阈值参数 | Lion | 热更新，不重启 |

## 五、真实项目中的协同案例

以"员工激励任务"业务为例：

```
1. Lion 配置任务参数（奖励金额、规则开关）
         ↓
2. Crane 每天 02:00 触发结算任务
         ↓
3. Squirrel 分布式锁防止重复结算
         ↓
4. Pigeon 调用积分服务发放奖励
         ↓
5. Mafka 发送"任务完成"消息
         ↓
6. 消费方更新用户通知、数据看板
```

五个框架缺一不可，各司其职。
