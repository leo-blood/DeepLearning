## 一、五大框架职责回顾

```
┌──────────────────────────────────────────────────────────────┐
│                    职责划分速查                                │
│                                                               │
│  Pigeon  → 同步调用其他服务（有返回值，需要立即知道结果）        │
│  Mafka   → 异步通知其他服务（不需要立即知道结果）              │
│  Crane   → 时间驱动的周期性操作                               │
│  Squirrel → 共享状态（缓存热数据、原子操作、分布式锁）          │
│  Lion    → 运行时可变的参数（开关、阈值、模板）                 │
└──────────────────────────────────────────────────────────────┘
```

**选型决策树：**

```
需要调用另一个服务？
  ├── 需要立即拿到结果 → Pigeon（同步 RPC）
  └── 不需要立即结果 → Mafka（异步消息）

需要定期执行？
  └── Crane（分布式定时任务）

需要共享数据？
  ├── 热点查询数据（缓存）→ Squirrel
  ├── 原子计数/限流 → Squirrel
  └── 防并发竞争 → Squirrel 分布式锁

需要参数可调整？
  └── Lion（配置中心）
```

## 二、综合案例：员工激励任务系统

### 2.1 业务流程说明

```
业务需求：
  每月定时给完成任务的员工发放积分奖励
  员工每天可领取一次日常任务
  积分由积分服务（另一个微服务）管理
  奖励规则由运营人员随时调整
```

### 2.2 完整系统架构

```
┌─────────────────────────────────────────────────────────────────┐
│  Lion：奖励规则配置                                              │
│   ccs.admin.reward.points.per.task = 10                         │
│   ccs.admin.reward.daily.limit = 3（每天最多领3个任务）          │
└──────────────────────┬──────────────────────────────────────────┘
                        │ 动态配置
                        ▼
┌─────────────────────────────────────────────────────────────────┐
│  Crane：月度结算任务（每月1日 02:00）                             │
│   扫描上月完成的任务 → 计算应发积分 → 批量发放                    │
└──────────────────────┬──────────────────────────────────────────┘
                        │ 触发结算
                        ▼
┌─────────────────────────────────────────────────────────────────┐
│  Squirrel：                                                      │
│   分布式锁（防止月结重复执行）                                    │
│   缓存员工积分余额（减少 DB 查询）                               │
└──────────────────────┬──────────────────────────────────────────┘
                        │ 调用积分服务
                        ▼
┌─────────────────────────────────────────────────────────────────┐
│  Pigeon：调用积分微服务                                          │
│   PointService.grantPoints(staffMis, points)                    │
└──────────────────────┬──────────────────────────────────────────┘
                        │ 积分发放后通知
                        ▼
┌─────────────────────────────────────────────────────────────────┐
│  Mafka：发布"积分发放完成"事件                                    │
│   → 通知服务：推送消息给员工                                      │
│   → 数据看板：更新统计                                           │
│   → 运营系统：记录发放明细                                       │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 核心代码实现

```java
/**
 * 月度积分结算任务
 * 展示五大框架协同工作
 */
@Component
@Slf4j
public class MonthlyRewardSettlementJob {

    // Lion：动态配置
    @Autowired
    private LionService lionService;

    // Squirrel：分布式锁
    @Autowired
    private SquirrelDistributedLock distributedLock;

    // Pigeon：调用积分服务
    @MdpPigeonClient(service = "point-service")
    private PointService pointService;

    // Mafka：发送事件
    @Autowired
    private MafkaClient mafkaClient;

    @Autowired
    private TaskRepository taskRepository;

    @Crane(
        name = "monthly-reward-settlement",
        cron = "0 0 2 1 * ?",   // 每月1日 02:00
        description = "月度激励任务积分结算",
        timeout = 3600
    )
    public void run() {
        // 1. Lion 读取配置（运营可随时调整）
        int pointsPerTask = lionService.getRewardPointsPerTask();
        int batchSize = lionService.getSettlementBatchSize();

        log.info("[结算任务] 开始，pointsPerTask={}, batchSize={}", pointsPerTask, batchSize);

        // 2. Squirrel 分布式锁（防止多实例重复结算）
        String lockKey = "settlement:monthly:" + YearMonth.now().minusMonths(1);
        boolean locked = distributedLock.tryLock(lockKey, 3600);
        if (!locked) {
            log.warn("[结算任务] 未获取锁，跳过（其他实例正在执行）");
            return;
        }

        try {
            doSettlement(pointsPerTask, batchSize);
        } finally {
            distributedLock.unlock(lockKey);
        }
    }

    private void doSettlement(int pointsPerTask, int batchSize) {
        YearMonth lastMonth = YearMonth.now().minusMonths(1);
        Long lastId = 0L;
        int totalSettled = 0;

        while (true) {
            // 3. 分页查询上月完成的任务（Crane 批量分页模式）
            List<CompletedTask> tasks = taskRepository
                    .findCompletedTasksLastMonth(lastMonth, lastId, batchSize);

            if (tasks.isEmpty()) break;

            for (CompletedTask task : tasks) {
                try {
                    settleOneTask(task, pointsPerTask);
                    totalSettled++;
                } catch (Exception e) {
                    log.error("[结算任务] 单任务失败: taskId={}", task.getId(), e);
                    // 单条失败不影响整体
                }
            }

            lastId = tasks.get(tasks.size() - 1).getId();
            if (tasks.size() < batchSize) break;
        }

        log.info("[结算任务] 完成，totalSettled={}", totalSettled);
    }

    @Transactional
    private void settleOneTask(CompletedTask task, int pointsPerTask) {
        // CAS 防重（幂等）
        boolean updated = taskRepository.markAsSettled(task.getId());
        if (!updated) {
            log.debug("任务已结算，跳过: taskId={}", task.getId());
            return;
        }

        // 4. Pigeon 调用积分服务（同步，需要知道是否成功）
        GrantPointsRequest request = GrantPointsRequest.builder()
                .staffMis(task.getStaffMis())
                .points(pointsPerTask)
                .reason("月度任务奖励")
                .bizId("task:" + task.getId())  // 幂等 key
                .build();

        try {
            GrantPointsResult result = pointService.grantPoints(request);
            if (!result.isSuccess()) {
                throw new BusinessException("积分服务返回失败: " + result.getMessage());
            }
        } catch (PigeonException e) {
            // RPC 失败：回滚本地状态，抛出异常触发事务回滚
            log.error("积分服务调用失败: taskId={}, staffMis={}", task.getId(), task.getStaffMis(), e);
            throw new RuntimeException("积分服务不可用", e);
        }

        // 5. Mafka 发送结算完成事件（异步，不阻塞主流程）
        RewardGrantedEvent event = RewardGrantedEvent.builder()
                .taskId(task.getId())
                .staffMis(task.getStaffMis())
                .points(pointsPerTask)
                .settlementMonth(YearMonth.now().minusMonths(1).toString())
                .eventId(UUID.randomUUID().toString())
                .build();

        mafkaClient.sendMessage("ccs_reward_granted", JSON.toJSONString(event));
        // 发消息失败不影响主流程（最终一致性）
    }
}
```

## 三、框架交互的关键设计点

### 3.1 一致性保证

```
问题：Pigeon 调用成功，Mafka 发消息失败 → 积分发了，但没有通知

解决方案（按重要性选择）：

场景A：通知不重要（允许丢失）
  → 捕获异常，记录日志，继续
  → Mafka 失败不影响主流程

场景B：通知重要（不能丢失）
  → Outbox 模式：将消息写入 DB（同一事务）
  → Crane 异步发送 DB 中的 pending 消息

场景C：强一致（积分和通知必须同时成功或失败）
  → 分布式事务（TCC/Saga）
  → 复杂度高，慎用
```

### 3.2 异常传播边界

```java
// 框架调用的异常边界设计

// Pigeon 调用：异常向上传播（同步调用，失败要感知）
try {
    pointService.grantPoints(request);
} catch (PigeonException e) {
    throw new ServiceException("积分服务不可用", e);  // 向上传播
}

// Mafka 发消息：异常吞掉（异步消息，失败不影响主流程）
try {
    mafkaClient.sendMessage(topic, body);
} catch (Exception e) {
    log.error("消息发送失败，不影响主流程", e);  // 吞掉，记录日志
}

// Squirrel 锁操作：根据业务决定
try {
    squirrelClient.set(key, value);
} catch (Exception e) {
    log.warn("缓存写入失败，降级处理", e);  // 缓存失败通常允许降级
}
```

## 四、全链路故障排查指南

### 4.1 故障排查流程

```
用户反馈"积分没到账"

Step 1: 查 Crane 执行记录
  → 任务有没有执行？
  → 执行结果是成功还是失败？

Step 2: 查任务 DB 状态
  → task.settled_status = ?
  → 是否被标记为 SETTLED？

Step 3: 查 Pigeon 调用日志（用 TraceId）
  → 积分服务是否被调用？
  → 调用是否成功？

Step 4: 查 Mafka 消费日志
  → 通知是否发出？
  → 消费者是否处理了？

Step 5: 查 Squirrel 锁状态
  → 是否因为锁未释放导致任务跳过？

Step 6: 查 Lion 配置
  → 是否有错误的功能开关导致跳过了某个分支？
```

### 4.2 常见故障模式

| 故障现象 | 排查方向 |
|---------|---------|
| Crane 任务有执行记录但未处理数据 | 查是否获取了分布式锁（Squirrel），是否提前返回 |
| Pigeon 调用超时 | 查对方服务 CPU/GC，查数据库慢查询，检查超时配置 |
| Mafka 消费延迟增大 | 查消费者 Lag，检查处理逻辑是否有慢操作，考虑扩容 |
| Redis 命中率低 | 查 Key 设计是否合理，TTL 是否过短，是否有缓存穿透 |
| Lion 配置变更不生效 | 检查 @MdpConfigChange 是否注册，查推送日志 |

## 五、系统设计方法论总结

### 5.1 框架选型决策

```
需求来了，问自己三个问题：

1. 这个操作需要同步还是异步？
   同步（需要立即返回结果）→ Pigeon
   异步（通知型、事件型）   → Mafka

2. 这个操作是否有并发竞争？
   共享数据 / 防重复 / 原子计数 → Squirrel

3. 这个参数是否需要运行时可调？
   阈值 / 开关 / 模板 → Lion

4. 这个操作是否需要周期性执行？
   定时任务 → Crane
```

### 5.2 可靠性设计原则

| 原则 | 实践 |
|------|------|
| **幂等优先** | 每个操作设计为可重复执行 |
| **防御性编程** | 不信任外部调用的返回值 |
| **失败降级** | 非核心依赖失败时有兜底 |
| **异步解耦** | 能用消息队列的地方不用同步调用 |
| **锁粒度最小化** | 分布式锁的范围越小越好 |
| **配置外化** | 可能变化的参数都放 Lion |

### 5.3 监控指标体系

```
每个框架的核心监控：

Pigeon:  P99 RT < timeout * 0.8
         错误率 < 0.1%
         熔断状态 = CLOSED

Mafka:   消费 Lag < 1000 条
         死信队列消息数 = 0

Crane:   任务执行成功率 = 100%
         执行时间 < timeout * 0.8

Squirrel: Redis 命中率 > 80%
          P99 RT < 5ms

Lion:    配置推送延迟 < 10s
```
