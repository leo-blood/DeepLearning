# Lion 架构设计与最佳实践

---

## 1. 配置中心的架构原理

### 1.1 Lion 系统架构

```
┌─────────────────────────────────────────────────────┐
│                 Lion 服务端                          │
│  配置存储（MySQL）  │  变更推送服务  │  权限管控       │
└─────────────────────────────────────────────────────┘
          ▲ 读写配置                  │ 推送变更
          │                          │
          │                          ▼
┌─────────────────────────────────────────────────────┐
│              Lion Agent（每台应用机器上）             │
│                                                     │
│  启动时：拉取 AppKey 下所有配置 → 注入 @MdpConfig 字段│
│  运行时：接收变更推送 → 更新字段（反射赋值）           │
│  断线时：本地缓存（降级）                            │
└─────────────────────────────────────────────────────┘
```

### 1.2 配置的存储与推送

```
Lion 控制台修改配置
  │
  ├─ 写入 MySQL（配置存储）
  ├─ 记录变更历史（可回滚）
  │
  ▼
变更推送服务
  │
  ├─ 向订阅该 AppKey 的所有机器推送变更通知
  │   (长连接/WebSocket)
  │
  ▼
各机器的 Lion Agent 接收到推送
  │
  ├─ 更新本地缓存
  ├─ 通过反射更新 @MdpConfig 字段
  │
  ▼
下次方法调用时，字段已是新值（秒级生效）
```

## 2. 本项目 LionService 设计分析

### 2.1 集中管理的设计决策

```java
// ccsadmin-infrastructure/proxy/LionService.java
// 位置特意放在 infrastructure 层，而不是 starter 层

// 好处：
// 1. 各层都可以注入（starter/application/cloudoffice 都用到）
// 2. 配置项一览无余（20+ 个配置都在这一个类）
// 3. 统一默认值管理

// 使用示例
@Autowired
private LionService lionService;

// 激励任务延迟分钟
int delayMinutes = Optional.ofNullable(lionService.incentiveDelayMinutes).orElse(0);

// 员工技能批量查询大小
int skillBatchSize = lionService.getEsStaffSkillAmount();   // @Data 生成 getter
```

### 2.2 直接在业务类使用 vs LionService

**放 LionService 的原则**：多处使用或跨层使用

```java
// 多处使用：放 LionService
@MdpConfig("default.es.staff.skill.amount")
public Integer esStaffSkillAmount = 1000;
// StaffPigeonService 和其他多处都用到

// 只有一处用：直接放业务类
// CloudOfficeStaffApplyService 里
@MdpConfig("CLOUD_OFFICE_APPLY_GRAY:ALL")
private String CLOUD_OFFICE_APPLY_GRAY = "";
// 只有云办公申请逻辑用，不值得放 LionService
```

## 3. 与 KMS 的协作

### 3.1 KMS 解决什么问题

```
Lion 控制台：开发/测试/运维都能看到配置值
→ 不能存密钥（AppSecret、数据库密码等）

KMS（Key Management Service）：
  - 加密存储密钥
  - 只在应用启动时解密一次，存在内存里
  - 控制台看到的是密文 $KMS{keyName}

应用启动时：
  $KMS{uacSecret} → KMS 服务 → 明文 secret → 注入到配置
```

### 3.2 本项目 KMS + Lion 协作示例

```java
// ApplicationLoader.java —— 应用启动时处理 KMS

public static void main(String[] args) throws KmsResultNullException {
    SpringApplication application = new SpringApplication(ApplicationLoader.class);

    // 从 KMS 解密，设置为启动属性
    Map<String, Object> props = new HashMap<>();
    props.put("uac.host",   Kms.getByNameWithCache(appKey, "uacHost"));   // 内网地址
    props.put("uac.appKey", Kms.getByNameWithCache(appKey, "uacAppKey")); // 密钥
    props.put("uac.secret", Kms.getByNameWithCache(appKey, "uacSecret")); // 密钥
    application.setDefaultProperties(props);

    application.run(args);
}
```

```properties
# application.properties 里 KMS 占位符
cs.cli.uac-secret=$KMS{essuacSecret}   # 启动时 MDP 自动解密
cs.cli.uacSecret1=$KMS{uacSecret}      # 同上

dx.ccs.robot.appSecret=$KMS{dxRobotAppSecret}   # 大象机器人密钥
```

### 3.3 配置分类决策树

```
这个配置值是什么？
  │
  ├── 密钥/密码/Token？
  │     └── 放 KMS（$KMS{keyName}）+ application.properties 引用
  │
  ├── 运行时可能变化的值？
  │     ├── 功能开关/灰度控制 → 放 Lion（@MdpConfig）
  │     ├── 阈值/批量大小    → 放 Lion（@MdpConfig）
  │     └── 运营文案/URL    → 放 Lion（@MdpConfig）
  │
  ├── 启动时确定，不运行时变化？
  │     └── 放 application.properties（或 env-specific properties）
  │
  └── 永远不变的？
        └── 放代码常量（static final）
```

## 4. 环境隔离

### 4.1 三个环境的配置隔离

```
Lion 控制台按环境隔离：
  AppKey: com.sankuai.csccratos.ccsadmin
    ├── prod 环境
    │     key: incentive.room.size = 200（真实生产值）
    │     key: CLOUD_OFFICE_APPLY_GRAY = ALL
    │
    ├── staging 环境
    │     key: incentive.room.size = 10（测试值）
    │     key: CLOUD_OFFICE_APPLY_GRAY = zhangsan,lisi（测试白名单）
    │
    └── dev 环境（可选）
          key: incentive.room.size = 5

应用根据运行环境（MdpContextUtils.getHostEnvStr()）读取对应环境的 Lion 配置
```

### 4.2 配置变更流程规范

```
标准流程：
  1. 在 staging 环境修改配置，验证功能
  2. 观察 staging 运行正常（5-10分钟）
  3. 在 prod 环境修改配置
  4. 观察 prod 监控指标（5分钟）
  5. 确认无异常，变更完成

紧急修复（线上出问题）：
  1. 直接在 prod 修改（没有时间走完整流程）
  2. 事后在 staging 同步修改（保持环境一致）
  3. 写变更记录
```

## 5. 常见问题与排查

### 5.1 配置不生效

```
排查步骤：
  1. 确认 key 拼写：Lion 控制台检索 key，确认存在
  2. 确认 AppKey：配置是否在正确的 AppKey 下
  3. 确认环境：是否在 prod/staging 正确的环境
  4. 查看应用日志：Lion Agent 是否有报错
  5. 等待：推送有 1~5 秒延迟
  6. 检查字段注解：@MdpConfig 是否正确导入
     import com.meituan.mdp.boot.starter.config.annotation.MdpConfig;  ← 正确
```

### 5.2 配置值解析失败

```java
// 症状：Integer 类型字段为 null，虽然 Lion 配置了值

// 原因1：Lion 配置值不是合法数字
// Lion value: "100a"（含非数字字符）
// 框架尝试转 Integer 失败 → 字段为 null

// 原因2：Map/数组格式错误
// Lion value: {1:7,2:3}（key 没有引号，非合法 JSON）
// 框架解析失败 → 字段为空 Map

// 防御写法：总是对可能为 null 的字段做判空
int pageSize = pageSize != null ? pageSize : 200;
// 或
int pageSize = Optional.ofNullable(lionService.roomSize).orElse(50);
```

### 5.3 配置变更引发的异常

```
场景：Lion 修改了文案模板，新模板的 %s 占位符数量和代码不匹配

// 旧模板（2个占位符）
"您的员工%s已申请，请于%s前审批"

// 代码
String.format(template, staffMis, deadline);   // 传了2个参数，正确

// 错误修改（改成3个占位符）
"您的员工%s已申请，请于%s前审批，理由：%s"

// 代码没变，还是传2个参数 → MissingFormatArgumentException！

解决：
  1. 修改 Lion 配置前，确认占位符数量与代码匹配
  2. 代码层面做防御处理（try-catch 文案格式化）
  3. 使用命名参数替代位置参数（更安全）
```

## 6. 最佳实践总结

### 6.1 @MdpConfig 使用规范

```java
// ✅ 好的写法
@MdpConfig("feature.enabled:false")   // key 语义清晰，有默认值
private Boolean featureEnabled;

@MdpConfig("batch.size:200")          // 数字型有合理默认值
private Integer batchSize;

// ❌ 不好的写法
@MdpConfig("a")                       // key 太短，无意义
private String a;

@MdpConfig("SOME_KEY")                // 没有默认值，Lion 挂了时为 null
private Integer someValue;            // 使用前需要每次判空

// ❌ 不该放 Lion 的内容
@MdpConfig("db.password")            // 密码不能放 Lion → 用 KMS
private String dbPassword;
```

### 6.2 Lion 配置管理规范

| 规范 | 说明 |
|------|------|
| key 命名语义清晰 | 一看就知道是什么：`incentive.room.size` 而不是 `size` |
| 所有配置有默认值 | Lion 挂了时不影响服务 |
| staging 先验证 | 不要直接改 prod |
| 变更要记录 | 写明改了什么、为什么改、谁审批 |
| 密钥走 KMS | 绝对不在 Lion 存密码/Token |
| 集中管理 | 多处用的配置放 LionService |
| 定期清理 | 已下线功能的配置要及时删除 |
