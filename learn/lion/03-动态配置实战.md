## 1. 功能开关（Feature Flag）

### 1.1 灰度控制（本项目实际用法）

```java
// CloudOfficeStaffApplyService.java

@MdpConfig("CLOUD_OFFICE_APPLY_GRAY:ALL")  // ALL = 全量放开
private String CLOUD_OFFICE_APPLY_GRAY = "";
private static final String DEFAULT_GRAY_ALL = "ALL";

public void createApply(CloudOfficeStaffApplyRequestVO request) {
    String currentUserMis = UserUtils.getUser().getLogin();  // 当前登录用户

    // 根据 Lion 配置决定是否允许申请
    if (!DEFAULT_GRAY_ALL.equals(CLOUD_OFFICE_APPLY_GRAY)) {
        // 灰度模式：只允许白名单内的员工
        String[] allowedList = CLOUD_OFFICE_APPLY_GRAY.split(",");
        boolean inWhiteList = Arrays.stream(allowedList)
            .map(String::trim)
            .anyMatch(mis -> mis.equals(currentUserMis));

        if (!inWhiteList) {
            throw new BusiException("功能灰度中，暂不开放");
        }
    }
    // ALL 模式：所有人都可以
    doCreateApply(request);
}
```

### 1.2 灰度发布完整流程

```
阶段1：内测（只有几个人可用）
  Lion 设置 CLOUD_OFFICE_APPLY_GRAY = "zhangsan,lisi,wangwu"
  效果：只有这3个人能申请，其他人报"功能灰度中"

阶段2：小范围（10%用户）
  Lion 设置 = "mis1,mis2,...（100个mis）"
  效果：100个员工可用，验证稳定性

阶段3：全量上线
  Lion 设置 = "ALL"
  效果：所有人可用

阶段4：如果出问题，紧急关闭
  Lion 设置 = ""（空字符串）
  效果：所有人都不能申请（紧急止损）

整个过程不需要重启服务，修改 Lion 配置秒级生效！
```

## 3. 消息文案动态管理

### 3.1 通知文案（本项目实际用法）

```java
// CloudOfficeStaffApplyService.java

// 发给主管的通知文案（支持占位符）
@MdpConfig("CLOUD_OFFICE_APPLY_LEADER_SUCCESS:" +
    "您管理的员工mis%s已报名混合办公，预计居家时间为%s，" +
    "可在[生产力管理系统-云客服管理-混合办公管理|%s]中查看。")
private String CLOUD_OFFICE_APPLY_LEADER_SUCCESS;

// 发给员工的通知文案
@MdpConfig("CLOUD_OFFICE_APPLY_SUCCESS_STAFF:" +
    "您已成功报名%s的居家办公，请于%s前领取摄像头，" +
    "如未在规定时间内领取摄像头，将取消本次居家办公。")
private String CLOUD_OFFICE_APPLY_SUCCESS_STAFF;

// 配置项（提前天数）
@MdpConfig("CLOUD_OFFICE_APPLY_SUCCESS_STAFF_DAYS_BEFORE:2")
private Integer CLOUD_OFFICE_APPLY_SUCCESS_STAFF_DAYS_BEFORE = 2;

// 使用
private void sendSuccessNotify(StaffDTO staff, String planDate) {
    String leaderMsg = String.format(
        CLOUD_OFFICE_APPLY_LEADER_SUCCESS,
        staff.getMis(), planDate, CLOUD_OFFICE_APPLY_LEADER_SUCCESS_URL
    );
    String staffMsg = String.format(
        CLOUD_OFFICE_APPLY_SUCCESS_STAFF,
        planDate,
        DateUtils.addDays(planDate, -CLOUD_OFFICE_APPLY_SUCCESS_STAFF_DAYS_BEFORE)
    );

    xmService.sendMessage(staff.getLeaderMis(), leaderMsg);
    xmService.sendMessage(staff.getMis(), staffMsg);
}
```

**文案修改无需部署**：
```
运营：通知文案需要调整措辞
→ 联系技术：修改 Lion 配置 CLOUD_OFFICE_APPLY_SUCCESS_STAFF
→ 修改后秒级生效
→ 不需要提需求 → 不需要开发改代码 → 不需要测试 → 不需要发版
```

## 4. A/B 测试配置

### 4.1 基于用户分组的 A/B

```java
@MdpConfig("feature.ab.ratio:50")      // A方案占比，默认50%
private Integer abRatio = 50;

public String getNotifyTemplate(String staffMis) {
    // 根据 mis 的 hash 值决定走哪个分支
    int hash = Math.abs(staffMis.hashCode()) % 100;

    if (hash < abRatio) {
        // A 方案（abRatio % 的用户）
        return "简短版通知文案...";
    } else {
        // B 方案（100-abRatio % 的用户）
        return "详细版通知文案...";
    }
}

// 灰度比例控制：
// Lion: feature.ab.ratio = 10  → 10% 用户走新方案
// 验证数据好 → Lion: feature.ab.ratio = 50  → 50% 用户
// 全量通过 → Lion: feature.ab.ratio = 100 → 全量新方案
```

## 6. Lion 配置变更的注意事项

### 6.1 配置热更新的线程安全

```java
// @MdpConfig 字段更新不是原子的
// 在极端情况下可能读到"新旧混合"的状态

// 场景：批量发消息时，文案配置被更新
// 前100条：用旧文案
// 后200条：用新文案（配置已更新）
// → 同一批任务，消息格式不一致

// 通常可以接受，但强一致性场景需注意
// 解决方案：在 Job 开始时把配置快照化
@Crane("someJob")
public void someJob(String param) {
    // 在任务开始时读取一次，整个任务过程用这一份
    String template = lionService.getNotifyTemplate();  // 快照
    String url = lionService.essInProgressUrl;          // 快照

    for (Staff staff : staffList) {
        String msg = String.format(template, staff.getMis(), url);
        sendMsg(msg);
    }
}
```

### 6.2 修改配置前的检查

```
修改 Lion 配置的注意事项：

1. 确认 key 拼写正确（Lion 不会提示 key 不存在）
2. 确认 value 格式正确（数组用逗号分隔，Map 用 JSON 格式）
3. 验证默认值是否合理（Lion 挂了时的兜底值）
4. 灰度配置：先在 staging 环境验证，再改 prod
5. 变更后观察：关注业务指标5-10分钟，确认无异常
```

### 6.3 Lion 配置回滚

```
如果修改了配置后发现问题：
  Lion 控制台 → 找到该 key → 查看变更历史 → 点击"回滚"
  → 秒级恢复到上个版本

这就是配置中心比代码修改的最大优势：
  代码修改 → 重新部署（10-30分钟）
  Lion 配置 → 回滚（秒级）
```
