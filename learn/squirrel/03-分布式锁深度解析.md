# Squirrel 分布式锁深度解析

> 本文聚焦：理解分布式锁的原理，以及本项目中的实现与注意事项。

## 1. 为什么单机锁不够用

### 1.1 单机锁（synchronized）的局限

```java
// 只在当前 JVM 进程内有效
public synchronized void createApply(ApplyRequest req) {
    // 同一台机器：只有一个线程能进入
    checkAndCreate(req);
}
```

```
机器A（JVM进程1）     机器B（JVM进程2）
    │                     │
    │ synchronized锁       │ synchronized锁（独立的！）
    │                     │
  线程1进入              线程2进入    ← 两台机器同时执行！锁无效！
```

**单机锁的致命问题**：每台机器的锁是独立的，跨机器没有任何约束。

### 1.2 分布式锁的原理

利用**所有机器共享同一个 Redis 实例**，通过 SETNX 原子操作实现互斥：

```
机器A                   Redis（共享）             机器B
  │                        │                        │
  │── SETNX "lock" ──────▶ │                        │
  │◀── true（成功，获锁）── │                        │
  │                        │                        │
  │                        │ ◀── SETNX "lock" ──── │
  │                        │ ──── false（失败）────▶ │
  │                        │                        │
  │ 执行业务逻辑            │                     机器B知道有人在处理
  │                        │                        │
  │── DEL "lock" ─────────▶│                        │
  │                        │
```

**Redis 是单线程处理命令**，SETNX 是原子操作，所以天然线程安全。

## 2. 本项目分布式锁实现

### 2.1 云办公申请防重复提交

```java
// CloudOfficeStaffApplyService.java

// 锁的设计参数
private String staffIdLockFormat = "staff_id_lock-%s";  // key 模板（以员工ID为粒度）
private int lockExpireInSeconds = 60 * 30;               // 30分钟过期

public void createApply(CloudOfficeStaffApplyRequestVO request) {
    StaffDTO staffDTO = getStaffInfo(request);

    // ① 构造锁 key（不同员工用不同的锁，互不影响）
    String lockKey = String.format(staffIdLockFormat, staffDTO.getId());

    // ② 尝试加锁（SETNX）
    boolean locked = squirrelOpService.setNtxString(
        lockKey,
        staffDTO.getId(),        // value（调试时可看到是谁持锁）
        lockExpireInSeconds      // 过期时间（防死锁）
    );

    if (!locked) {
        // ③ 加锁失败 → 同一员工有请求正在处理
        log.info("员工{}申请处理中，拒绝重复提交, staffId:{}", staffDTO.getMis(), staffDTO.getId());
        throw new BusiException("您的申请正在处理中，请勿重复提交");
    }

    try {
        // ④ 加锁成功，执行业务
        doCreateApply(staffDTO, request);
    } finally {
        // ⑤ 无论成功失败，释放锁（finally 保证一定执行）
        try {
            squirrelOpService.delString(lockKey);
            log.info("锁已释放, staffId:{}", staffDTO.getId());
        } catch (Exception e) {
            log.warn("锁释放失败（过期后自动释放）, lockKey:{}", lockKey, e);
        }
    }
}
```

## 3. 分布式锁的三大问题

### 3.1 问题一：死锁（锁永不释放）

```
场景：加锁成功 → 程序崩溃 → 没有执行 DEL → 锁永远存在 → 其他请求永远被阻塞

解决：SETNX 时必须设置过期时间
      即使程序崩溃，锁也会在过期后自动释放

本项目设置：30分钟 = 业务最长处理时间的2倍（申请处理通常10-15分钟）
```

```java
// ✅ 必须设置过期时间
storeClient.setnx(storeKey, value, 60 * 30);

// ❌ 不设置过期时间 = 可能死锁
storeClient.set(storeKey, value);  // 没有过期时间！
```

### 3.2 问题二：误释放（A 释放了 B 的锁）

```
场景：
  T=0: 线程A 加锁，key="lock"，expire=30min
  T=1: 线程A 处理业务（超时，未完成）
  T=30min: 锁到期自动释放
  T=31min: 线程B 加锁成功，key="lock"
  T=32min: 线程A 终于处理完，执行 DEL "lock" → 删掉了线程B的锁！
  T=33min: 线程C 加锁，认为没有锁，进来了 → 线程B和C同时在执行！
```

**解决方案**：锁的 value 存储唯一标识（如 UUID），释放前验证 value：

```java
// 更安全的锁实现（本项目使用了员工ID作为value，有一定保护）
String lockValue = UUID.randomUUID().toString();  // 唯一标识

// 加锁时存入唯一标识
storeClient.setnx(lockKey, lockValue, expireSeconds);

// 释放锁时：先检查是否是自己的锁
String currentValue = storeClient.get(lockKey);
if (lockValue.equals(currentValue)) {
    storeClient.delete(lockKey);  // 只删自己的锁
} else {
    log.warn("锁已过期被其他线程持有，不删除");
}
```

**本项目的实际做法**：用员工 ID 作为 value，但没有做误释放保护（value 验证）。在云办公申请这个场景，业务上同一员工的申请是顺序处理的，即使锁过期也不会并发，所以可接受。

### 3.3 问题三：锁粒度太粗导致性能下降

```
// ❌ 太粗：锁住所有员工的申请
String lockKey = "cloud_office_apply_global_lock";

// 后果：1000 个员工同时申请 → 全部排队，串行化处理

// ✅ 细粒度：每个员工一把锁
String lockKey = "staff_id_lock-" + staffDTO.getId();

// 效果：1000 个员工各自持有自己的锁，互不影响，并行处理
```

## 4. SETNX 的底层原子性

### 4.1 为什么 SETNX 是线程安全的

```
Redis 执行命令的线程模型：

客户端A: SETNX lock value ─────────┐
客户端B: SETNX lock value ─────────┤
客户端C: SETNX lock value ─────────┘
                                    │
                              Redis 单线程队列
                                    │
                          ┌─── 依次处理 ───┐
                          │               │
                      客户端A: 成功     客户端B: 失败
                                         客户端C: 失败

Redis 是单线程处理命令（I/O多路复用），
SETNX 执行过程中不会被其他命令打断，天然原子性。
```

### 4.2 SETNX vs SET NX（Redis 命令区别）

```redis
# 旧命令（需要两步，非原子）
SETNX key value          # 设置
EXPIRE key 60            # 设置过期 → 这两步之间可能崩溃！

# 新命令（一步，原子）
SET key value NX EX 60   # NX=不存在时设置，EX=过期时间

# Squirrel 的 setnx 方法内部使用的是新的原子命令
storeClient.setnx(storeKey, value, expireSeconds)
// 等价于：SET key value NX EX expireSeconds
```

## 5. 完整的分布式锁使用规范

### 5.1 加锁到释放的完整流程

```java
String lockKey = "your_business_lock:" + bizId;
int expireSeconds = 300;   // 5分钟（根据业务耗时设置）

boolean locked = squirrelOpService.setNtxString(lockKey, bizId, expireSeconds);

if (!locked) {
    // 加锁失败的处理
    // 选项1：直接拒绝（本项目的做法）
    throw new BusiException("操作进行中，请勿重复提交");

    // 选项2：等待重试（适合短暂等待的场景）
    // int maxRetry = 3;
    // while (!locked && maxRetry-- > 0) {
    //     Thread.sleep(1000);
    //     locked = squirrelOpService.setNtxString(lockKey, bizId, expireSeconds);
    // }
    // if (!locked) throw new BusiException("系统繁忙，请稍后重试");
}

try {
    // 业务逻辑
    doYourBusiness();
} finally {
    // 释放锁（finally 保证必须执行）
    try {
        squirrelOpService.delString(lockKey);
    } catch (Exception e) {
        log.warn("锁释放失败，将等待自动过期, key:{}", lockKey, e);
    }
}
```

### 5.2 锁设计检查清单

```
☑ 锁 key 设计：包含业务维度（员工ID/任务ID），不能太粗（全局锁）
☑ 过期时间：不能省略，值 = 业务处理最长时间 × 2
☑ finally 释放：必须在 finally 块中释放锁
☑ 加锁失败处理：明确告知用户当前状态
☑ 释放锁异常处理：释放失败时打 warn 日志（不影响业务返回）
☑ 幂等保护：即使锁失效导致并发，业务逻辑本身也要有防重
```
