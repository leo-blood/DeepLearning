## 1. 缓存三大问题

### 1.1 缓存穿透（Cache Penetration）

**问题**：查询一个**不存在的 key**（如员工 mis="INVALID"），每次都打到数据库：

```
流程：
  请求 GET staff:INVALID
  → 缓存未命中（key不存在）
  → 查数据库：没有这条数据
  → 不写缓存（没数据可写）
  → 下次请求还是穿透到数据库

恶意攻击：大量不存在的 key → 打垮数据库
```

**解决**：缓存空值

```java
public StaffDTO getStaffInfo(String mis) {
    String cacheKey = "staff:" + mis;
    String cachedValue = squirrelOpService.getString(cacheKey);

    if ("NULL".equals(cachedValue)) {
        return null;   // 缓存的空值，直接返回 null，不查 DB
    }
    if (cachedValue != null) {
        return JSON.parseObject(cachedValue, StaffDTO.class);
    }

    // 查数据库
    StaffDTO staff = staffMapper.queryByMis(mis);

    if (staff == null) {
        // 缓存空值，短暂时间（防恶意攻击导致大量空值占内存）
        squirrelOpService.setString(cacheKey, "NULL", 60);  // 60秒
    } else {
        squirrelOpService.setString(cacheKey, JSON.toJSONString(staff), 300);
    }

    return staff;
}
```

### 1.2 缓存击穿（Cache Breakdown）

**问题**：某个**热点 key**过期，大量并发请求同时打到数据库：

```
T=0: staff:zhangsan 缓存过期
T=0.001: 100个并发请求同时查询 staff:zhangsan
T=0.001: 100个请求都缓存未命中
T=0.001: 100个请求都去查数据库 → 数据库瞬间高压
```

**解决**：用 SETNX 做"单飞"（只允许一个请求查数据库）

```java
public StaffDTO getStaffInfoWithSingleFlight(String mis) {
    String cacheKey = "staff:" + mis;
    String lockKey = "staff_rebuild_lock:" + mis;

    // 先查缓存
    StaffDTO cached = squirrelOpService.getString(cacheKey);
    if (cached != null) return cached;

    // 缓存未命中，用锁保证只有一个请求查数据库
    boolean locked = squirrelOpService.setNtxString(lockKey, "1", 5);   // 5秒锁

    if (locked) {
        try {
            // 获锁成功：查数据库并重建缓存
            StaffDTO fresh = staffMapper.queryByMis(mis);
            if (fresh != null) {
                squirrelOpService.setString(cacheKey, fresh, 300);
            }
            return fresh;
        } finally {
            squirrelOpService.delString(lockKey);
        }
    } else {
        // 获锁失败：等一下，让获锁的请求先建好缓存，再从缓存读
        Thread.sleep(50);
        return squirrelOpService.getString(cacheKey);
    }
}
```

### 1.3 缓存雪崩（Cache Avalanche）

**问题**：大量 key **同时**过期，请求同时打到数据库：

```
T=0: 批量写入缓存（假设都是600秒过期）
T=600: 所有 key 同时过期
T=600.001: 所有请求同时打到数据库 → 雪崩！
```

**解决**：过期时间加随机扰动

```java
// ✅ 在基础过期时间上加随机值（±10%）
public void setStaffCache(String mis, StaffDTO staff) {
    int baseExpire = 300;
    // 加随机扰动：270~330 秒之间
    int expire = baseExpire + new Random().nextInt(60) - 30;
    squirrelOpService.setString("staff:" + mis, staff, expire);
}

// ✅ 永不过期 + 后台刷新（适合超热点数据）
// 数据本身不设过期，另起线程定期更新
```

## 2. 分布式 ID 生成

### 2.1 用 INCRBY 生成分布式序列号

```java
// 批量获取 ID 段，减少 Redis 调用次数
public List<Long> generateBatchIds(String bizType, int count) {
    String idKey = "global_id:" + bizType;

    // 原子加 count，获取 [result-count+1, result] 的 ID 段
    Long maxId = squirrelOpService.incrBy(idKey, count, 86400 * 365 * 10);  // 10年过期

    List<Long> ids = new ArrayList<>();
    for (long i = maxId - count + 1; i <= maxId; i++) {
        ids.add(i);
    }
    return ids;
}

// 使用
List<Long> newIds = generateBatchIds("incentive_task", 100);   // 一次获取100个ID
```

## 3. 防重复提交

### 3.1 表单重复提交防护

本项目的实际场景（云办公申请）已经实现，可以推广到其他场景：

```java
// 通用的防重提交工具方法
public boolean tryAcquireSubmitLock(String bizType, String bizId, int expireSeconds) {
    String lockKey = String.format("%s_submit_lock:%s", bizType, bizId);

    boolean acquired = squirrelOpService.setNtxString(
        lockKey, System.currentTimeMillis(), expireSeconds
    );

    if (!acquired) {
        log.info("重复提交，拒绝, bizType:{}, bizId:{}", bizType, bizId);
    }
    return acquired;
}

public void releaseSubmitLock(String bizType, String bizId) {
    String lockKey = String.format("%s_submit_lock:%s", bizType, bizId);
    try {
        squirrelOpService.delString(lockKey);
    } catch (Exception e) {
        log.warn("释放提交锁失败, key:{}", lockKey, e);
    }
}

// 使用
public void createCloudOfficeApply(CloudOfficeStaffApplyRequestVO req) {
    String staffId = getCurrentStaffId();
    if (!tryAcquireSubmitLock("cloudoffice_apply", staffId, 1800)) {
        throw new BusiException("申请处理中，请勿重复提交");
    }
    try {
        doCreateApply(req);
    } finally {
        releaseSubmitLock("cloudoffice_apply", staffId);
    }
}
```

## 4. 监控与内存管理

### 4.1 Redis 内存使用注意事项

```
本项目 category: cratos-supplement
Redis 实例：cscRedis

风险：
  - 缓存的 key 越来越多，内存增长
  - 没有过期时间的 key 永不释放

规范：
  ✅ 所有 key 必须设置过期时间
  ✅ 分布式锁过期时间不超过业务最长处理时间
  ❌ 不允许永久存储业务数据（数据应放 DB）
```

## 5. 最佳实践总结

### 5.1 Squirrel 操作规范清单

```
Key 设计：
  ☑ 包含业务命名空间（通过 category 区分）
  ☑ key 格式：{业务}:{实体}:{ID}
  ☑ 所有 key 有明确的过期时间

缓存操作：
  ☑ 读缓存异常时降级查 DB（不抛异常）
  ☑ 写缓存异常时只打 warn 日志（不影响业务）
  ☑ 删缓存异常时只打 warn 日志（等自然过期）

分布式锁：
  ☑ 过期时间设置为业务最长处理时间 × 2
  ☑ finally 块中释放锁
  ☑ 加锁失败明确提示用户
  ☑ 锁 key 粒度精准（不能太粗）

计数器：
  ☑ 使用 incrBy/decrBy，不用 get+set（非原子）
  ☑ 设置合理过期时间（统计周期对应）
```

### 5.2 Squirrel vs 数据库的选择

| 场景 | 选 Squirrel | 选 MySQL |
|------|------------|---------|
| 分布式锁 | ✅ | 可以，但复杂 |
| 热点数据缓存 | ✅ | ❌（太慢）|
| 计数器（不需精确） | ✅ | 可以 |
| 需要持久化的业务数据 | ❌ | ✅ |
| 需要事务的操作 | ❌ | ✅ |
| 历史记录/审计 | ❌ | ✅ |
| 复杂查询（JOIN） | ❌ | ✅ |
