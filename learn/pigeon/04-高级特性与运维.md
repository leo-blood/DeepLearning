# Pigeon 高级特性与运维

## 1. 负载均衡与服务路由

### 1.1 负载均衡策略

Consumer 向 MNS 查询到多台 Provider 机器时，Pigeon 默认使用**加权轮询**：

```
Provider 机器列表（来自 MNS）：
  10.0.1.23:5001 （权重 1）
  10.0.1.24:5001 （权重 1）
  10.0.1.25:5001 （权重 1）

请求路由：
  请求1 → 10.0.1.23
  请求2 → 10.0.1.24
  请求3 → 10.0.1.25
  请求4 → 10.0.1.23（轮回）
  ...
```

### 1.2 本地调用优先（同机房优化）

同一机房的机器调用延迟更低，Pigeon 支持本地优先策略：

```
机房A（上海）：
  ccsadmin 机器 → 优先调用同机房的 MDM 机器
  万不得已才跨机房调用

机房B（北京）：
  ccsadmin 机器 → 优先调用机房B的 MDM 机器
```

这一策略由框架自动处理，开发者无需关心。

## 2. 熔断与降级

### 2.1 什么是熔断

```
正常状态：
  Consumer → Provider （Provider 正常响应）

异常累积（Provider 出问题）：
  Consumer → Provider 超时/报错
  Consumer → Provider 超时/报错
  Consumer → Provider 超时/报错（连续失败N次）
               ↓
           熔断器打开

熔断状态（保护 Provider）：
  Consumer → 直接返回错误（不再调用 Provider）
  （经过一段时间后，熔断器进入半开状态，试探 Provider 是否恢复）
```

### 2.2 代码层面的降级处理

Pigeon 框架有熔断机制，但代码层面也必须做降级：

```java
// StaffPigeonService.java 的降级模式
public Optional<StaffDTO> getStaffByStaffMis(String staffMis) {
    if (staffMis == null) return Optional.empty();

    try {
        BaseResultDTO<StaffDTO> res = iStaffRemoteService.queryByLoginNameFromCache(staffMis);
        return Optional.ofNullable(res.isSuccess() ? res.getData() : null);

    } catch (Exception e) {
        // 无论是超时、熔断、网络错误，都降级返回空
        log.error("查询员工失败（降级），staffMis:{}", staffMis, e);
        return Optional.empty();   // ← 降级值，不影响主流程
    }
}
```

**降级策略选择**：

| 场景 | 降级策略 |
|------|---------|
| 查询类（员工信息、技能标签）| 返回空，跳过该员工的处理 |
| 核心写操作（扣减权益）| 不降级，直接报错，让用户重试 |
| 非关键通知（发大象消息）| 静默失败，只记录日志 |

## 3. 链路追踪与监控

### 3.1 CAT 链路追踪

本项目的日志前缀：

```java
// CommonConstant.java
public static final String LOG_PREFIX_CLOUD_CUSTOMER_SERVICE_PLATFORM = "CCS_PLATFORM";

// 所有 Pigeon 相关日志加前缀，方便在 CAT 上按业务搜索
log.error("{}, checkTrainingPacakge error, packageId:{}",
    CommonConstant.LOG_PREFIX_CLOUD_CUSTOMER_SERVICE_PLATFORM, packageId, e);
```

在 CAT 监控平台，可以追踪一个请求从：
```
HTTP 接口 → Service → Pigeon 调用（MDM）→ 返回 → 响应
```
每一步的耗时、成功率都可以看到。

### 3.2 关键监控指标

| 指标 | 说明 | 告警阈值（参考） |
|------|------|----------------|
| 调用成功率 | `success / total` | < 99% 告警 |
| P99 响应时间 | 99% 请求的响应时间 | > 超时时间 × 80% |
| 超时率 | 超时次数 / 总次数 | > 1% 告警 |
| QPS | 每秒调用次数 | 异常波动 |

## 4. 版本兼容与接口升级

### 4.1 兼容升级（推荐方式）

```java
// 旧版接口（已有调用方在用）
public interface IncentiveService {
    ResultVO<Boolean> checkTrainingPacakge(Long packageId);
}

// 升级：增加新接口，保留旧接口（调用方平滑迁移）
public interface IncentiveService {
    // 保留旧接口
    ResultVO<Boolean> checkTrainingPacakge(Long packageId);

    // 新增接口（支持更多参数）
    ResultVO<CheckPackageResult> checkTrainingPacakgeV2(CheckPackageRequest request);
}
```

### 4.2 接口下线流程

```
步骤1：确认没有调用方再使用旧接口（查 CAT 监控）
步骤2：在接口上加 @Deprecated 注解
步骤3：通知所有使用方迁移到新接口
步骤4：观察1-2个月，确认旧接口调用量为0
步骤5：从接口定义和实现中删除旧接口，升版本号
```

## 5. 本地开发与测试

### 5.1 本地无法访问 MNS 怎么办

美团内部的 MNS 在内网，本地开发一般无法直接访问真实的 Pigeon 服务。

常用方案：
```
方案1：连 VPN 访问测试环境（staging 环境）
  MdpContextUtils.getHostEnvStr() = "staging"
  连接 staging 环境的 MNS，调用 staging 的 MDM

方案2：Mock 外部 Pigeon 服务
  @MockBean
  IStaffPigeonService iStaffPigeonService;  // 单测时 Mock

方案3：本地启动 Provider 服务（极少用）
```

### 5.2 单测 Mock 示例

```java
@SpringBootTest
public class StaffPigeonServiceTest {

    @Autowired
    private StaffPigeonService staffPigeonService;

    @MockBean
    IStaffRemoteService iStaffRemoteService;   // Mock 外部服务

    @Test
    public void testGetStaffByMis() {
        // Given: 模拟 MDM 返回数据
        StaffDTO mockStaff = new StaffDTO("zhangsan", "张三");
        BaseResultDTO<StaffDTO> mockResult = BaseResultDTO.success(mockStaff);
        Mockito.when(iStaffRemoteService.queryByLoginNameFromCache("zhangsan"))
               .thenReturn(mockResult);

        // When
        Optional<StaffDTO> result = staffPigeonService.getStaffByStaffMis("zhangsan");

        // Then
        assertTrue(result.isPresent());
        assertEquals("zhangsan", result.get().getLoginName());
    }

    @Test
    public void testGetStaffByMis_Exception() {
        // 模拟 RPC 异常
        Mockito.when(iStaffRemoteService.queryByLoginNameFromCache(any()))
               .thenThrow(new RuntimeException("Pigeon timeout"));

        // 验证降级行为
        Optional<StaffDTO> result = staffPigeonService.getStaffByStaffMis("zhangsan");
        assertFalse(result.isPresent());   // 降级返回空，不抛异常
    }
}
```

## 6. 常见问题汇总

### 6.1 启动时注册失败

```
现象：应用启动日志有 "Pigeon server register failed"

排查：
  1. 检查网络：是否能访问 MNS（内网问题）
  2. 检查 AppKey：是否在 MNS 有注册权限
  3. 检查端口：5001 端口是否被占用（netstat -tlnp | grep 5001）

影响：对外暴露的 Pigeon 服务不可用，但应用本身能启动
```

### 6.2 调用方 NoSuchBeanDefinitionException

```
现象：@MdpPigeonClient 注入时，应用启动失败

排查：
  1. 检查接口类是否在 classpath 中（api jar 是否引入）
  2. 检查 @MdpPigeonClient 注解是否正确导入（com.meituan.mdp.boot...）
  3. 检查字段类型是否是接口而不是实现类
```

### 6.3 数据不一致（反序列化异常）

```
现象：调用成功但返回的对象字段为空，或序列化报错

排查：
  1. 确认 Consumer 和 Provider 使用的 API jar 版本一致
  2. 确认 DTO 类有无参构造函数（Hessian 要求）
  3. 确认没有使用不可序列化的字段类型（如 LocalDateTime 需要特殊处理）
```

### 6.4 本项目 Pigeon Server 端口

```properties
# pigeon.properties
pigeon.server.port=5001      # Pigeon 服务监听端口
pigeon.server.http.port=5002 # Pigeon HTTP 协议端口（调试用）
```
