# Pigeon 服务端开发实战

> 本文聚焦：如何让本项目（ccsadmin）对外暴露 Pigeon 服务，供其他系统调用。

---

## 1. 服务端开发三步走

```
Step 1: api 模块定义接口（其他系统引用这个 jar）
Step 2: starter 模块实现接口 + @MdpPigeonServer
Step 3: 应用启动后自动注册到 MNS，其他系统即可调用
```

---

## 2. Step 1：在 api 模块定义接口

### 2.1 接口放哪里

```
ccsadmin-api/
└── src/main/java/
    └── .../api/
        └── service/
            └── IncentiveService.java   ← Pigeon 接口定义在这里
```

`ccsadmin-api` 模块会被打成独立 jar（版本号 `api.version=1.0.39`），外部系统通过 Maven 引用：

```xml
<!-- 外部系统的 pom.xml -->
<dependency>
    <groupId>com.sankuai.csc.cratos</groupId>
    <artifactId>ccsadmin-api</artifactId>
    <version>1.0.39</version>
</dependency>
```

### 2.2 接口设计规范

```java
// IncentiveService.java

// ✅ 好的接口设计
public interface IncentiveService {

    /**
     * 检查培训内容包是否可删除
     * 业务语义：内容包有进行中的激励任务时，不允许删除
     *
     * @param packageId 内容包ID，不能为 null
     * @return ResultVO<Boolean> true=可删除，false=不可删除
     */
    ResultVO<Boolean> checkTrainingPacakge(Long packageId);
}
```

**接口设计要点**：

| 要点 | 说明 | 示例 |
|------|------|------|
| 返回 ResultVO 包装 | 统一错误码，不抛异常 | `ResultVO<Boolean>` |
| 参数用包装类型 | 避免基本类型序列化歧义 | `Long` 不用 `long` |
| 方法语义清晰 | 名字说明做什么，不说怎么做 | `checkTrainingPacakge` |
| 不依赖内部 BO/DO | 参数/返回值只用 api 模块自己定义的 DTO | `ResultVO` 在 api 模块 |

### 2.3 ResultVO 结构

```java
// 本项目的统一返回包装类
public class ResultVO<T> {
    private boolean success;
    private String errorCode;
    private String errorMsg;
    private T data;

    // 成功
    public static <T> ResultVO<T> newSuccessObject(T data) { ... }

    // 业务错误（如参数不合法）
    public static <T> ResultVO<T> newBusiErrorObject(String msg) { ... }

    // 系统错误（未预期异常）
    public static <T> ResultVO<T> newServerErrorObject() { ... }
}
```

## 3. Step 2：实现接口 + @MdpPigeonServer

### 3.1 最简实现

```java
// IncentiveServiceImpl.java

@MdpPigeonServer      // ← 关键注解，应用启动时自动注册到 MNS
@Slf4j
public class IncentiveServiceImpl implements IncentiveService {

    @Autowired
    private IncentiveMapperService incentiveMapperService;

    @Override
    public ResultVO<Boolean> checkTrainingPacakge(Long packageId) {
        try {
            List<IncentiveTask> tasks = incentiveMapperService
                .queryUnFinishTaskByContent(IncentiveTaskTypeEnum.CONTENT_TASK.getCode(), packageId);
            return ResultVO.newSuccessObject(CollectionUtils.isEmpty(tasks));
        } catch (Exception e) {
            log.error("checkTrainingPacakge error, packageId:{}", packageId, e);
            return ResultVO.newServerErrorObject();
        }
    }
}
```

### 3.2 @MdpPigeonServer 的本质

`@MdpPigeonServer` 告诉 MDP：
1. 这个 Bean 实现了某个接口（IncentiveService）
2. 在服务端口（默认 5001）监听针对这个接口的调用
3. 向 MNS 注册本机地址和接口名
4. 收到请求时反序列化参数，调用此 Bean 的方法

**不加 `@MdpPigeonServer` 的后果**：接口实现了，但没有注册到 MNS，其他系统调用时找不到服务 → `ServiceUnavailableException`

## 4. 异常处理策略（重点！）

### 4.1 为什么不能抛异常

Pigeon 服务接口**绝对不能**把业务异常直接抛出：

```java
// ❌ 错误写法
public ResultVO<Boolean> checkTrainingPacakge(Long packageId) {
    // 如果这里抛出异常
    List<IncentiveTask> tasks = incentiveMapperService.query(packageId);
    return ResultVO.newSuccessObject(tasks.isEmpty());
}

// 后果：
// Consumer 收到的不是 ResultVO，而是一个 RPC 调用异常（PigeonException）
// Consumer 无法区分"业务失败"和"网络/系统故障"
// Consumer 可能把系统异常当业务异常处理，逻辑出错
```

### 4.2 标准的异常处理模板

```java
@Override
public ResultVO<Boolean> checkTrainingPacakge(Long packageId) {

    // 1. 入参校验
    if (packageId == null || packageId <= 0) {
        return ResultVO.newBusiErrorObject("packageId 不合法");
    }

    try {
        // 2. 业务逻辑
        List<IncentiveTask> tasks = incentiveMapperService
            .queryUnFinishTaskByContent(CONTENT_TASK.getCode(), packageId);

        boolean canDelete = CollectionUtils.isEmpty(tasks);
        log.info("checkTrainingPacakge packageId:{}, canDelete:{}", packageId, canDelete);
        return ResultVO.newSuccessObject(canDelete);

    } catch (BusiException e) {
        // 3. 预期的业务异常
        log.warn("checkTrainingPacakge busi error, packageId:{}, msg:{}", packageId, e.getMessage());
        return ResultVO.newBusiErrorObject(e.getMessage());

    } catch (Exception e) {
        // 4. 未预期的系统异常
        log.error("{}, checkTrainingPacakge system error, packageId:{}",
            CommonConstant.LOG_PREFIX_CLOUD_CUSTOMER_SERVICE_PLATFORM, packageId, e);
        return ResultVO.newServerErrorObject();   // ← 不暴露异常细节给外部
    }
}
```

### 4.3 异常处理分类

| 异常类型 | 处理方式 | 返回什么 |
|---------|---------|---------|
| 入参不合法 | 直接 return | `newBusiErrorObject("参数错误")` |
| 业务规则不满足（BusiException）| catch，warn 日志 | `newBusiErrorObject(e.getMessage())` |
| 数据库查询失败 | catch，error 日志 | `newServerErrorObject()` |
| RPC 调用失败 | catch，error 日志 | `newServerErrorObject()` |
| NullPointerException | catch，error 日志 | `newServerErrorObject()` |

## 6. 版本兼容性管理

### 6.1 接口升级原则

```
场景：需要给 checkTrainingPacakge 增加一个参数 taskType

❌ 错误做法（破坏性修改）：
  旧：ResultVO<Boolean> checkTrainingPacakge(Long packageId)
  新：ResultVO<Boolean> checkTrainingPacakge(Long packageId, Integer taskType)
  → 老版本 Consumer 调用新版本 Provider → 序列化错误

✅ 正确做法（向后兼容）：
  Option 1：新增一个方法
    ResultVO<Boolean> checkTrainingPacakge(Long packageId)              // 保留
    ResultVO<Boolean> checkTrainingPacakgeV2(Long packageId, Integer taskType) // 新增

  Option 2：使用 DTO 参数（可扩展）
    原始接口：checkTrainingPacakge(Long packageId)
    改为：    checkTrainingPacakge(CheckPackageRequest req)
    // CheckPackageRequest 以后加字段，不影响方法签名
```

### 6.2 API 版本号

```xml
<!-- pom.xml -->
<api.version>1.0.39</api.version>

<!-- 修改接口时需要升版本号 -->
<!-- 不兼容修改（改参数/删方法） → 升主版本：1.x.x → 2.x.x -->
<!-- 新增方法 → 升次版本：1.0.x → 1.1.x -->
<!-- Bug 修复 → 升修订版本：1.0.38 → 1.0.39 -->
```

```java
// 入口日志（可选，高频接口注意性能）
log.info("checkTrainingPacakge called, packageId:{}", packageId);

// 关键决策日志（必须）
log.info("checkTrainingPacakge packageId:{}, unfinishedTaskCount:{}, canDelete:{}",
    packageId, tasks.size(), canDelete);

// 警告日志（业务异常）
log.warn("checkTrainingPacakge busi error, packageId:{}, msg:{}", packageId, e.getMessage());

// 错误日志（系统异常，必须带异常栈）
log.error("{}, checkTrainingPacakge error, packageId:{}",
    CommonConstant.LOG_PREFIX_CLOUD_CUSTOMER_SERVICE_PLATFORM, packageId, e);
//         ↑ 固定前缀，用于日志检索                                          ↑ 带 Throwable 打栈
```
